Требования заинтересованных сторон

Модуль справочной БД для ПО AlphaMeterQC

Автор: В.Л. Солодюк Проект: ПО «AlphaMeterQC» (анализ и контроль своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии и хранящихся в БД (Oracle) ПО АльфаЦЕНТР) / Модуль справочной БД

Термины и определения

Термин Определение
Система Разрабатываемый модуль справочной базы данных для хранения атрибутов точек учёта, используемый совместно с ПО AlphaMeterQC.
Точка учёта (ТУ) Именованная точка учёта электрической энергии, идентифицируемая в пределах объекта/электроустановки уникальным номером присоединения/фидера в БД АльфаЦЕНТР.
Объект Объект/электроустановка, в пределах которого осуществляется учёт электрической энергии. Идентифицируется уникальным номером в БД АльфаЦЕНТР.
Потребитель Организация, потребляющая электрическую энергию, учёт которой осуществляется через принадлежащие ей точки учёта.
Статус ТУ Категория точки учёта (ОРЭМ, МТП, ДСД и др.), каждый из которых характеризуется максимально допустимым временем отсутствия поступления данных в БД АльфаЦЕНТР.
Справочная БД Разрабатываемая локальная база данных, размещаемая на сервере в локальной сети предприятия, для хранения справочной информации о точках учёта.
БД АльфаЦЕНТР База данных Oracle программного комплекса «АльфаЦЕНТР», содержащая оперативные данные со счётчиков электрической энергии.

Заинтересованные стороны

ID Заинтересованное лицо/сторона Описание
S-1 Инженеры отдела системного сопровождения Основные конечные пользователи модуля. Ежедневно работают с ПО AlphaMeterQC для контроля данных со счётчиков. Добавляют, актуализируют и удаляют справочную информацию.
S-2 Руководитель отдела системного сопровождения Внутренний заказчик. Отвечает за эффективность работы отдела, KPI по времени формирования отчётности и точности данных.
S-3 Команда разработки ПО AlphaMeterQC Технические специалисты, которые будут интегрировать справочную БД в основное приложение.
S-4 Администраторы серверов локальной сети Специалисты, обеспечивающие развёртывание, поддержку и резервное копирование справочной БД.
S-5 Отдел информационной безопасности Контролирующий орган. Следит за соблюдением политик разграничения доступа и защиты данных.

Требования заинтересованных сторон

1. Состав и структура справочных данных

ID Заинтересованное лицо Версия Содержание требования
SR-1 S-1, S-2 v1 Система должна обеспечивать хранение информации об организациях, включая их наименование. Организации могут являться Потребителями.
SR-2 S-1, S-2 v1 Система должна обеспечивать хранение информации о структурных подразделениях организаций, включая наименование подразделения и его принадлежность к организации. Каждая организация может иметь одно и более структурных подразделений.
SR-3 S-1, S-2 v1 Система должна обеспечивать хранение информации о контактных лицах из числа работников структурных подразделений, включая: фамилию, имя, отчество, должность, контактный телефон и электронную почту. Каждое структурное подразделение может иметь ноль или более контактных лиц.
SR-4 S-1, S-2 v1 Система должна обеспечивать хранение информации о местах расположения объектов, включая наименование места расположения. В одном месте расположения может находиться более одного объекта.
SR-5 S-1, S-2 v1 Система должна обеспечивать хранение информации об объектах/электроустановках, включая: наименование объекта, уникальный номер объекта в БД АльфаЦЕНТР, место расположения и структурное подразделение, в эксплуатационной ответственности которого находится объект. Каждый объект относится ровно к одному месту расположения и ровно к одному структурному подразделению.
SR-6 S-1, S-2 v1 Система должна обеспечивать хранение информации о точках учёта (ТУ), включая: наименование ТУ, уникальный номер присоединения/фидера в БД АльфаЦЕНТР, принадлежность к объекту и статус. Каждая ТУ относится ровно к одному объекту.
SR-7 S-1, S-2 v1 Система должна обеспечивать хранение справочника статусов ТУ, включая: наименование статуса (ОРЭМ, МТП, ДСД и др.) и максимально допустимое время отсутствия поступления данных в БД АльфаЦЕНТР (в часах).
SR-8 S-1, S-2 v1 Система должна обеспечивать хранение информации о заявках по ТУ, включая: номер заявки, дату начала и дату окончания. Одна ТУ может иметь несколько заявок.
SR-9 S-1, S-2 v1 Система должна обеспечивать хранение текстовых примечаний по каждой ТУ. Одна ТУ может иметь несколько примечаний.

2. Управление справочными данными

ID Заинтересованное лицо Версия Содержание требования
SR-10 S-1 v1 Система должна предоставлять возможность добавления новых записей во все справочники (организации, подразделения, контактные лица, места расположения, объекты, ТУ, статусы, заявки, примечания).
SR-11 S-1 v1 Система должна предоставлять возможность редактирования существующих записей во всех справочниках.
SR-12 S-1 v1 Система должна предоставлять возможность удаления записей из справочников с обеспечением контроля ссылочной целостности (запрет удаления записей, на которые имеются ссылки).
SR-13 S-1 v1 Система должна обеспечивать целостность связей между сущностями: при удалении организации должны быть обработаны все связанные подразделения; при удалении объекта — все связанные ТУ; при удалении ТУ — все связанные заявки и примечания.

3. Интеграция с ПО AlphaMeterQC

ID Заинтересованное лицо Версия Содержание требования
SR-14 S-3 v1 Система должна предоставлять программный интерфейс (API) для выполнения операций чтения и записи справочных данных из ПО AlphaMeterQC.
SR-15 S-3 v1 Система должна обеспечивать возможность выполнения SQL-запросов для комбинирования данных справочной БД с данными БД АльфаЦЕНТР при формировании экранных форм.
SR-16 S-3 v1 Система должна использовать открытый стандартизированный диалект SQL для обеспечения совместимости с различными СУБД.

4. Производительность и надёжность

ID Заинтересованное лицо Версия Содержание требования
SR-17 S-1, S-2 v1 Время выполнения типового запроса на чтение справочных данных (список ТУ по объекту, список заявок по ТУ) не должно превышать 1 секунды при объёме данных до 10 000 записей.
SR-18 S-4 v1 Система должна обеспечивать стабильную работу в круглосуточном режиме 24/7 с доступностью не менее 99,5%.
SR-19 S-4 v1 Система должна поддерживать процедуры резервного копирования и восстановления данных без остановки работы основного приложения.

5. Развёртывание и администрирование

ID Заинтересованное лицо Версия Содержание требования
SR-20 S-4 v1 Система должна быть развёрнута на сервере в локальной сети предприятия с использованием реляционной СУБД с открытым исходным кодом (PostgreSQL или SQLite).
SR-21 S-4 v1 Система должна поддерживать многопользовательский режим работы с возможностью одновременного подключения нескольких АРМ инженеров.
SR-22 S-4 v1 Процесс развёртывания БД должен быть документирован и включать скрипты создания схемы данных, начального наполнения справочников и настройки прав доступа.

6. Требования информационной безопасности

ID Заинтересованное лицо Версия Содержание требования
SR-23 S-5 v1 Система должна обеспечивать разграничение прав доступа к данным: чтение доступно всем авторизованным пользователям, запись (добавление, изменение, удаление) — только уполномоченным сотрудникам.
SR-24 S-5 v1 Система должна вести журнал изменений данных (аудит) с фиксацией: кто, когда и какие изменения внёс в справочную информацию.
SR-25 S-5 v1 Передача данных между справочной БД и АРМ инженеров должна осуществляться по защищённому протоколу в рамках корпоративной сети.
