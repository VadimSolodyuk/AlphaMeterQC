Анализ и устранение узких мест в производительности справочной БД AlphaMeterQC

Автор: В.Л. Солодюк Проект: ПО «AlphaMeterQC» (анализ и контроль своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии и хранящихся в БД (Oracle) ПО АльфаЦЕНТР) / Модуль справочной БД

1. Назначение документа

Настоящий документ содержит анализ узких мест в производительности справочной БД AlphaMeterQC, выявленных в ходе нагрузочного тестирования (Приложение О), а также рекомендации по их устранению. Документ разработан в рамках этапа «Оптимизация и тестирование БД» (раздел 4 отчёта).

2. Методика анализа

Анализ узких мест проводился в следующем порядке:

  1. Сбор данных о производительности — на основе результатов нагрузочного тестирования (10 сценариев, Приложение О) и оптимизации запросов (8 типовых запросов, Приложение Н).
  2. Идентификация потенциальных узких мест — выявление запросов с наибольшим временем выполнения, операций полного сканирования таблиц (Seq Scan), неоптимальных планов выполнения.
  3. Оценка влияния — определение критичности каждого узкого места для эксплуатации БД при текущих и прогнозируемых объёмах данных.
  4. Разработка рекомендаций — предложения по устранению или смягчению выявленных узких мест.

4. Идентифицированные узкие места

4.1. Узкое место №1: Полное сканирование таблицы ТочкаУчёта в агрегирующем запросе (T4)

Описание. Запрос статистики ТУ по статусам (T4) выполняет полное сканирование (Seq Scan) таблицы ТочкаУчёта при 10 000 записей. Время выполнения — 2,168 мс, количество прочитанных буферов — 45.

План выполнения (10 000 записей):

Hash Right Join (actual time=0.018..1.245 rows=10000 loops=1)
  ->  Seq Scan on ТочкаУчёта tu (actual time=0.005..0.512 rows=10000 loops=1)
        Buffers: shared hit=45
  ->  Hash (actual time=0.004..0.004 rows=6 loops=1)
        ->  Seq Scan on СтатусТУ s (actual time=0.002..0.002 rows=6 loops=1)

Причина. Для агрегирующего запроса, которому необходимо прочитать все строки таблицы, оптимизатор PostgreSQL обоснованно выбирает полное сканирование вместо индексного. Seq Scan читает всю таблицу последовательно, что для 10 000 строк эффективнее, чем random access через индекс с последующим чтением каждой строки.

Критичность. Низкая. Время 2,168 мс значительно ниже порога в 1 000 мс. Однако при росте объёма данных до 100 000 записей и более время выполнения будет расти линейно.

Прогноз при росте объёма:

Объём ТУ Прогнозируемое время (Seq Scan) Запас до порога 1 000 мс
10 000 2,2 мс 454×
50 000 ~11 мс 90×
100 000 ~22 мс 45×
500 000 ~110 мс 9×
1 000 000 ~220 мс 4,5×

Рекомендации по устранению:

  1. Материализованное представление (Materialized View) — создать материализованное представление, которое будет хранить предрассчитанную статистику и обновляться по расписанию или после массовых изменений:
CREATE MATERIALIZED VIEW справочная_бд.мв_статистика_ту_по_статусам AS
SELECT
    s.ИД_Статуса,
    s.Наименование AS Статус,
    COUNT(tu.ИД_ТУ) AS КоличествоТУ,
    s.МаксВремяОтсутствияДанныхЧас
FROM справочная_бд.СтатусТУ s
    LEFT JOIN справочная_бд.ТочкаУчёта tu ON s.ИД_Статуса = tu.ИД_Статуса
GROUP BY s.ИД_Статуса, s.Наименование, s.МаксВремяОтсутствияДанныхЧас
ORDER BY s.Наименование;

-- Индекс для быстрого доступа
CREATE UNIQUE INDEX idx_мв_статистика_статус
    ON справочная_бд.мв_статистика_ту_по_статусам(ИД_Статуса);

-- Обновление по расписанию (ежедневно)
-- REFRESH MATERIALIZED VIEW справочная_бд.мв_статистика_ту_по_статусам;
  1. Покрывающий индекс — если требуется выполнять запрос без полного сканирования, можно создать покрывающий индекс, включающий все необходимые столбцы:
CREATE INDEX idx_ту_статус_covering
    ON справочная_бд.ТочкаУчёта(ИД_Статуса) INCLUDE (ИД_ТУ);

Однако для агрегирующего запроса, читающего все строки, покрывающий индекс не даст значительного выигрыша по сравнению с Seq Scan.

4.2. Узкое место №2: Отсутствие композитного индекса для иерархического запроса (T2)

Описание. Запрос полной иерархии (T2) соединяет 5 таблиц последовательными вложенными циклами (Nested Loop). При текущем объёме данных (10 000 ТУ) время выполнения составляет 0,112 мс, что является отличным показателем. Однако при росте объёма данных до 50 000–100 000 ТУ количество итераций вложенных циклов может привести к увеличению времени.

План выполнения (10 000 ТУ):

Nested Loop (actual time=0.035..0.078 rows=2 loops=1)
  ->  Nested Loop (actual time=0.024..0.038 rows=2 loops=1)
        ->  Index Scan on Организация (pk) — 1 строка
        ->  Nested Loop (actual time=0.010..0.019 rows=2 loops=1)
              ->  Index Scan on СтруктурноеПодразделение (idx_подразделение_организация)
              ->  Index Scan on Объект (idx_объект_подразделение)
  ->  Index Scan on ТочкаУчёта (idx_ту_объект)
  ->  Index Scan on СтатусТУ (pk)

Причина. Каждый уровень иерархии добавляет вложенный цикл. Хотя каждый цикл использует индексный поиск, при большом количестве строк на каждом уровне (например, 100 подразделений у организации, 10 объектов у подразделения, 10 ТУ у объекта) количество итераций растёт мультипликативно.

Критичность. Низкая. При текущем объёме данных (10 000 ТУ) время 0,112 мс не является проблемой.

Прогноз при росте объёма:

Сценарий Текущее время Прогноз при 100 000 ТУ Запас до порога 1 000 мс
T2 (полная иерархия) 0,112 мс ~1–2 мс 500×

Рекомендации по устранению:

  1. Композитный индекс (как рекомендовано в Приложении Н, раздел 4.1) для покрытия соединения ТочкаУчёта → СтатусТУ:
CREATE INDEX idx_ту_объект_статус
    ON справочная_бд.ТочкаУчёта(ИД_Объекта, ИД_Статуса);

Этот индекс позволит выполнять Index Only Scan для соединения ТУ со статусом, без обращения к таблице.

  1. При достижении 50 000 записей — пересмотреть план выполнения и при необходимости добавить композитные индексы для других уровней иерархии.

4.3. Узкое место №3: Отсутствие индекса для фильтрации заявок по дате (T5)

Описание. Запрос T5 выполняет поиск заявок по ТУ с фильтром по дате окончания. При текущем объёме (30 000 заявок) запрос использует Index Scan по idx_заявка_ту с последующей фильтрацией (Filter) по условию даты. Время — 0,022 мс.

План выполнения:

Index Scan using idx_заявка_ту on Заявка z
    (cost=0.29..8.57 rows=2 width=46) (actual time=0.010..0.013 rows=2 loops=1)
  Index Cond: (ИД_ТУ = 1)
  Filter: ((ДатаОкончания IS NULL) OR (ДатаОкончания >= '2026-01-01'::date))

Причина. Индекс idx_заявка_ту покрывает только условие ИД_ТУ = ?. Фильтрация по дате выполняется как дополнительное условие (Filter) после индексного поиска. При большом количестве заявок на одну ТУ (например, 100+) фильтрация по дате будет выполняться в памяти для каждой найденной строки.

Критичность. Низкая. При текущем объёме данных время 0,022 мс не является проблемой.

Рекомендации по устранению:

  1. Композитный индекс для покрытия обоих условий:
CREATE INDEX idx_заявка_ту_дата
    ON справочная_бд.Заявка(ИД_ТУ, ДатаОкончания);

Этот индекс позволит выполнять Index Scan с двумя условиями (Index Cond) без дополнительной фильтрации.

  1. Частичный индекс для часто используемого фильтра (только открытые заявки):
CREATE INDEX idx_заявка_ту_открытые
    ON справочная_бд.Заявка(ИД_ТУ)
    WHERE ДатаОкончания IS NULL;

4.4. Узкое место №4: Каскадное удаление при массовых операциях (T10)

Описание. При удалении ТУ (T10) выполняются каскадные операции: удаление связанных заявок и примечаний через триггеры внешних ключей. При текущем объёме время удаления одной ТУ — 0,312 мс, из которых 0,033 мс приходится на каскадные триггеры.

План выполнения:

Delete on ТочкаУчёта (actual time=0.245..0.245 rows=0 loops=1)
  ->  Index Scan using pk_точкаучёта (actual time=0.010..0.011 rows=1 loops=1)
  Trigger for constraint fk_заявка_ту: time=0.018 calls=1
  Trigger for constraint fk_примечание_ту: time=0.015 calls=1

Причина. Каскадное удаление требует поиска и удаления связанных записей в таблицах Заявка и Примечание. При массовом удалении (например, удаление объекта со всеми ТУ) количество каскадных операций растёт пропорционально количеству ТУ.

Критичность. Средняя. При единичных операциях время незначительно. При массовом удалении (например, удаление организации с каскадным удалением всех связанных записей) время может стать заметным.

Прогноз при массовом удалении:

Операция Количество каскадных записей Прогнозируемое время
Удаление 1 ТУ ~3 заявки + ~3 примечания 0,3 мс
Удаление объекта (10 ТУ) ~30 заявок + ~30 примечаний ~3 мс
Удаление подразделения (100 ТУ) ~300 заявок + ~300 примечаний ~30 мс
Удаление организации (500 ТУ) ~1 500 заявок + ~1 500 примечаний ~150 мс

Рекомендации по устранению:

  1. Индексы на внешних ключах — уже созданы (idx_заявка_ту, idx_примечание_ту), что обеспечивает эффективный поиск связанных записей при каскадном удалении. Дополнительное индексирование не требуется.

  2. Пакетное удаление — при необходимости массового удаления рекомендуется выполнять операцию в транзакции с предварительным удалением дочерних записей:

BEGIN;
DELETE FROM справочная_бд.Примечание WHERE ИД_ТУ IN (SELECT ИД_ТУ FROM справочная_бд.ТочкаУчёта WHERE ИД_Объекта = 1);
DELETE FROM справочная_бд.Заявка WHERE ИД_ТУ IN (SELECT ИД_ТУ FROM справочная_бд.ТочкаУчёта WHERE ИД_Объекта = 1);
DELETE FROM справочная_бд.ТочкаУчёта WHERE ИД_Объекта = 1;
COMMIT;
  1. Отложенное каскадное удаление — для очень больших объёмов (более 10 000 записей) рассмотреть возможность использования пакетной обработки с фиксацией транзакции после каждого пакета (например, по 1 000 записей).

4.5. Узкое место №5: Разрастание мёртвых кортежей при интенсивных DML-операциях

Описание. PostgreSQL использует механизм MVCC (Multi-Version Concurrency Control), при котором UPDATE и DELETE не изменяют существующие строки, а создают новые версии (кортежи). Старые версии помечаются как «мёртвые» и подлежат удалению процессом autovacuum. При интенсивных DML-операциях (частые обновления статусов ТУ, массовые вставки заявок) количество мёртвых кортежей может расти быстрее, чем autovacuum успевает их обрабатывать.

Причина. Механизм MVCC PostgreSQL. Каждая операция UPDATE создаёт новую версию строки; старая версия остаётся до обработки autovacuum.

Критичность. Низкая — средняя (зависит от интенсивности DML-операций).

Рекомендации по устранению:

  1. Настройка autovacuum для таблиц с высокой интенсивностью DML:
-- Для таблицы ТочкаУчёта (частые обновления статусов)
ALTER TABLE справочная_бд.ТочкаУчёта SET (
    autovacuum_vacuum_scale_factor = 0.01,    -- Срабатывание при 1% мёртвых кортежей
    autovacuum_vacuum_threshold = 50,          -- Минимальный порог мёртвых кортежей
    autovacuum_analyze_scale_factor = 0.005,   -- Анализ при 0.5% изменений
    autovacuum_vacuum_cost_limit = 1000        -- Увеличение интенсивности vacuum
);

-- Для таблицы Заявка (частые вставки и обновления)
ALTER TABLE справочная_бд.Заявка SET (
    autovacuum_vacuum_scale_factor = 0.02,
    autovacuum_vacuum_threshold = 100,
    autovacuum_analyze_scale_factor = 0.01
);
  1. Мониторинг мёртвых кортежей — регулярный контроль с помощью запроса:
SELECT
    relname AS Таблица,
    n_dead_tup AS МёртвыеКортежи,
    n_live_tup AS ЖивыеКортежи,
    round(n_dead_tup * 100.0 / GREATEST(n_live_tup + n_dead_tup, 1), 1) AS ПроцентМёртвых
FROM pg_stat_user_tables
WHERE schemaname = 'справочная_бд'
ORDER BY n_dead_tup DESC;
  1. Ручной VACUUM при превышении 20 % мёртвых кортежей:
VACUUM (VERBOSE, ANALYZE) справочная_бд.ТочкаУчёта;

4.6. Узкое место №6: Отсутствие мониторинга производительности

Описание. В текущей конфигурации не настроен мониторинг производительности БД. При росте объёма данных или изменении характера нагрузки ухудшение производительности может быть обнаружено только по жалобам пользователей.

Критичность. Средняя. Отсутствие мониторинга увеличивает время реакции на проблемы производительности.

Рекомендации по устранению:

  1. Базовый мониторинг с использованием системных представлений PostgreSQL:
-- Топ-10 самых медленных запросов (требуется pg_stat_statements)
SELECT
    queryid,
    round(total_exec_time::numeric, 2) AS ОбщееВремяМс,
    calls AS КоличествоВызовов,
    round(mean_exec_time::numeric, 2) AS СреднееВремяМс,
    round(max_exec_time::numeric, 2) AS МаксВремяМс,
    query
FROM pg_stat_statements
WHERE query NOT LIKE '%pg_stat%'
ORDER BY total_exec_time DESC
LIMIT 10;
  1. Мониторинг размера таблиц для отслеживания роста данных:
SELECT
    relname AS Таблица,
    pg_size_pretty(pg_total_relation_size(relid)) AS ОбщийРазмер,
    pg_size_pretty(pg_relation_size(relid)) AS РазмерТаблицы,
    pg_size_pretty(pg_indexes_size(relid)) AS РазмерИндексов,
    n_live_tup AS КоличествоЗаписей
FROM pg_catalog.pg_statio_user_tables
WHERE schemaname = 'справочная_бд'
ORDER BY pg_total_relation_size(relid) DESC;
  1. Рекомендуемые пороги оповещения:
Метрика Порог Действие
Время выполнения запроса > 500 мс Анализ плана выполнения
Количество мёртвых кортежей > 20 % VACUUM
Размер таблицы > 1 GB Рассмотреть партиционирование
Размер индекса > 500 MB Рассмотреть REINDEX
Свободное место на диске < 20 % Расширение дискового пространства

5. Матрица узких мест

№ Узкое место Текущее влияние Прогнозируемое влияние Критичность Приоритет устранения
1 Seq Scan в агрегирующем запросе (T4) 2,2 мс (10 000 записей) ~220 мс (1 000 000 записей) Низкая Низкий
2 Отсутствие композитного индекса (T2) 0,1 мс (10 000 записей) ~2 мс (100 000 записей) Низкая Низкий
3 Отсутствие индекса по дате (T5) 0,02 мс (30 000 заявок) ~0,5 мс (300 000 заявок) Низкая Низкий
4 Каскадное удаление (T10) 0,3 мс (1 ТУ) ~150 мс (500 ТУ) Средняя Средний
5 Разрастание мёртвых кортежей Не проявляется При интенсивных DML Средняя Средний
6 Отсутствие мониторинга Не проявляется При росте объёмов Средняя Высокий

6. Выводы

  1. Критических узких мест не выявлено. Все типовые запросы и операции DML выполняются за время, значительно ниже установленных требований (запас прочности от 460× до 3 800×).

  2. Потенциальные узкие места носят прогнозируемый характер и проявятся только при значительном росте объёма данных (от 100 000 записей и более).

  3. Рекомендации по устранению подготовлены для каждого узкого места и могут быть реализованы по мере роста объёма данных:

  4. Материализованное представление для агрегирующих запросов (узкое место №1);
  5. Композитные индексы для иерархических запросов (узкое место №2, №3);
  6. Пакетное удаление для массовых операций (узкое место №4);
  7. Настройка autovacuum для таблиц с интенсивными DML (узкое место №5);

  8. Внедрение базового мониторинга производительности (узкое место №6).

  9. Наивысший приоритет имеет внедрение мониторинга производительности (узкое место №6), так как без него невозможно своевременно обнаружить деградацию производительности при росте объёмов данных.
