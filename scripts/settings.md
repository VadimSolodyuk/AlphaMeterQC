## Анализ текущей системы промптов

Сейчас у вас есть:

- **Главный промпт** (`00_ai_assistant.md`) — содержит `important_rules`, делегирование ролям, правила передачи контекста.
- **Ролевые промпты** (`01_ba`, `02_architect`, `03_developer`, `04_qa`, `05_devops`) — каждый со своими обязанностями, правилами, чек-листами.
- **Вспомогательные промпты** (`context_transfer`, `feature_workflow`, `project_structure`) — вызываются по `@имя`.

Проблемы, которые вы заметили:

1. **Неясно, когда и какой контекст загружать.** AI должен сам решать, какие промпты включить в свой ответ. Если он включает все — перерасход токенов. Если не включает нужные — теряется качество.
2. **Механизм «вызова промпта» не определён.** `@ИмяРоли` — это просто ссылка или команда вставить содержимое? В текущей реализации, вероятно, AI просто читает файл промпта и использует его правила, но не всегда ясно, нужно ли дублировать текст в ответе.
3. **Передача контекста между ролями** описана, но триггеры размыты. Когда AI должен сформировать `context_transfer`? После каждого ответа? Только при завершении этапа?
4. **Дублирование информации.** Например, `feature_workflow` дублирует этапы из `workflow-template.md`. `context_transfer` — отдельный промпт, хотя мог бы быть частью `stage-output-template.md`.

## Предложения по оптимизации

### 1. Ввести «мета-промпт» для управления контекстом

Создайте файл `.continue/prompts/assistant/context_management.md` (или добавьте секцию в `00_ai_assistant.md`), который будет содержать **единственную таблицу принятия решений**:

```markdown
## Правила выбора контекста

При получении запроса:
1. Определи тип задачи по таблице ниже.
2. Загрузи **только** соответствующие промпты (по имени файла).
3. Если задача затрагивает несколько ролей — загрузи все, но используй их последовательно.
4. **Не включай полный текст промптов в ответ** — применяй их правила неявно, если не требуется явная цитата.
5. Если нужна структура проекта — используй `@project_structure` (он короткий).
6. Если нужно передать контекст — сформируй блок `[CONTEXT_TRANSFER]` по шаблону (см. ниже).

| Тип задачи | Необходимые промпты | Доп. контекст |
|------------|---------------------|---------------|
| Бизнес-анализ (требования, use cases, SRS) | `01_ba` | `project_structure` (если нужно понять модули) |
| Архитектурное проектирование (ADR, NFR) | `02_architect` | `01_ba` (если есть требования) |
| Реализация кода | `03_developer` | `02_architect` (если есть ADR), `project_structure` |
| Тестирование | `04_qa` | `03_developer` (код), `01_ba` (граничные случаи) |
| Инфраструктура/DevOps | `05_devops` | `project_structure`, `03_developer` (зависимости) |
| Передача контекста между ролями | `context_transfer` (шаблон) | — |
| Полный цикл фичи | `feature_workflow` + все роли по порядку | — |
```

**Преимущество:** AI сразу знает, какие промпты ему нужны, и не загружает лишние.

### 2. Унифицировать `context_transfer` и `stage-output-template`

Сейчас `context_transfer` — отдельный промпт, а `stage-output-template.md` содержит блок `[CONTEXT_TRANSFER]`. Логично **объединить**: сделать `context_transfer` частью шаблона stage-output, а не отдельным промптом. Тогда AI будет формировать stage-output в конце каждого этапа, и в нём уже будет блок передачи контекста.

**Изменение:** удалить `context_transfer.md`, а в `stage-output-template.md` добавить явные правила, когда и как заполнять блок. Например:

```markdown
## Передача контекста
Заполняется **только** если этап завершён и есть следующая роль.
- role: [BA/Architect/Developer/QA/DevOps]
- reason: краткое описание
- files: список изменённых файлов
- details: ключевые решения, граничные случаи
```

Тогда AI будет вызывать `@stage_output` (или просто следовать шаблону) вместо отдельного `@context_transfer`.


### 4. Сделать `project_structure` лениво загружаемым

Сейчас `project_structure` — это отдельный промпт, который можно вызвать. Но AI может сам решать, нужен ли он. Добавить правило:

> Если запрос касается конкретного файла или модуля, используй `grep_search` или `read_file_range` для получения структуры, а не полный `project_structure`. Полный `project_structure` вызывай только при первом знакомстве с проектом или при запросе общей структуры.

### 7. Пример оптимизированного workflow

1. Пользователь: «Спроектируй архитектуру для модуля X».
2. AI определяет тип: архитектурное проектирование → загружает `02_architect` (и, возможно, `01_ba`, если есть требования).
3. AI не включает полный текст промптов в ответ, а применяет их правила.
4. После завершения AI формирует stage-output с блоком `[CONTEXT_TRANSFER]` для Developer.
5. Если пользователь продолжает с Developer, AI видит блок и использует его как контекст.

## Конкретные изменения в файлах

### Ролевые промпты
- Вынести общие разделы в `common_rules.md` (или в `00_ai_assistant.md`).
- Оставить только уникальные обязанности и чек-листы.

### `feature_workflow.md`
- Сократить до краткой схемы, сославшись на `workflow-template.md` для деталей.

## Итог

После этих изменений:
- AI будет точно знать, какие промпты ему нужны для каждой задачи.
- Контекст будет минимальным (только необходимые правила).
- Передача контекста станет предсказуемой и автоматической.
- Общий объём промптов уменьшится за счёт устранения дублирования.
