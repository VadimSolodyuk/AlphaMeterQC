## 2.2 Проектирование логической модели базы данных

На основе концептуальной ER-диаграммы (пункт 2.1, Приложение Г) и требований технического задания (Приложение Д) была разработана логическая модель базы данных. Логическая модель является промежуточным звеном между концептуальной моделью (сущности и связи) и физической реализацией в СУБД PostgreSQL: она описывает структуру данных в терминах реляционной модели — таблицы, столбцы, типы данных, ограничения целостности, первичные и внешние ключи, индексы — без привязки к конкретным особенностям реализации СУБД.

Полная спецификация логической модели приведена в Приложении Е «Логическая модель базы данных» (Приложение Е). Ниже приведено описание ключевых проектных решений.

**Методология проектирования.** Проектирование логической модели выполнялось в соответствии с классической реляционной теорией баз данных и включало следующие этапы:

1) **Преобразование сущностей в таблицы.** Каждая сущность концептуальной модели (Организация, СтруктурноеПодразделение, КонтактноеЛицо, МестоРасположения, Объект, ТочкаУчёта, СтатусТУ, Заявка, Примечание) отображена в отдельную таблицу реляционной схемы.

2) **Определение первичных ключей.** Для каждой таблицы определён суррогатный первичный ключ — целочисленный идентификатор с автоматической генерацией значений (GENERATED ALWAYS AS IDENTITY). Применение суррогатных ключей обосновано следующими соображениями:
   а) естественные ключи (например, номер присоединения для ТУ) могут изменяться в процессе эксплуатации;
   б) целочисленные ключи обеспечивают максимальную производительность при соединении таблиц и построении индексов;
   в) единообразие именования (`ИД_<Сущность>`) упрощает написание и сопровождение SQL-запросов.

3) **Определение внешних ключей и правил ссылочной целостности.** Для каждой связи между сущностями определён внешний ключ с явно заданным правилом каскадного удаления в соответствии с бизнес-семантикой:
   а) **CASCADE** — для связей, где дочерняя сущность не имеет самостоятельного смысла без родительской (Организация → Подразделение, Подразделение → КонтактноеЛицо, Объект → ТочкаУчёта, ТочкаУчёта → Заявка, ТочкаУчёта → Примечание);
   б) **RESTRICT** — для связей, где удаление родительской записи должно быть запрещено при наличии дочерних (Подразделение → Объект, МестоРасположения → Объект, СтатусТУ → ТочкаУчёта).

4) **Определение типов данных и ограничений.** Для каждого столбца определён тип данных (INTEGER, VARCHAR, BOOLEAN, DATE, TIMESTAMP, TEXT) и ограничения целостности:
   а) NOT NULL — для обязательных атрибутов;
   б) UNIQUE — для атрибутов, уникальных по бизнес-логике (НомерПрисоединения, НомерВБДАльфаЦЕНТР, Наименование статуса);
   в) DEFAULT — для атрибутов со значениями по умолчанию (ПризнакПотребителя = FALSE, ДатаСоздания = CURRENT_TIMESTAMP);
   г) CHECK — для проверки допустимых значений (МаксВремяОтсутствияДанныхЧас > 0).

5) **Проектирование индексов.** Для обеспечения производительности типовых запросов спроектированы индексы по внешним ключам и часто используемым в поиске атрибутам. Всего определено 10 индексов (Приложение Е, раздел 4).

**Спецификация таблиц логической модели.** Логическая модель включает 9 таблиц, каждая из которых соответствует сущности концептуальной модели:

1. **Таблица `Организация`** — 3 столбца: ИД_Организации (PK), Наименование (NOT NULL), ПризнакПотребителя (DEFAULT FALSE).
2. **Таблица `СтруктурноеПодразделение`** — 3 столбца: ИД_Подразделения (PK), ИД_Организации (FK → Организация, CASCADE), Наименование (NOT NULL).
3. **Таблица `КонтактноеЛицо`** — 7 столбцов: ИД_КонтактногоЛица (PK), ИД_Подразделения (FK → СтруктурноеПодразделение, CASCADE), Фамилия (NOT NULL), Имя, Отчество, Должность, Телефон, Email.
4. **Таблица `МестоРасположения`** — 2 столбца: ИД_МестаРасположения (PK), Наименование (NOT NULL).
5. **Таблица `Объект`** — 5 столбцов: ИД_Объекта (PK), Наименование (NOT NULL), НомерВБДАльфаЦЕНТР (NOT NULL, UNIQUE), ИД_МестаРасположения (FK → МестоРасположения, RESTRICT), ИД_Подразделения (FK → СтруктурноеПодразделение, RESTRICT).
6. **Таблица `ТочкаУчёта`** — 5 столбцов: ИД_ТУ (PK), Наименование (NOT NULL), НомерПрисоединения (NOT NULL, UNIQUE), ИД_Объекта (FK → Объект, CASCADE), ИД_Статуса (FK → СтатусТУ, RESTRICT).
7. **Таблица `СтатусТУ`** — 3 столбца: ИД_Статуса (PK), Наименование (NOT NULL, UNIQUE), МаксВремяОтсутствияДанныхЧас (NOT NULL, CHECK > 0).
8. **Таблица `Заявка`** — 5 столбцов: ИД_Заявки (PK), ИД_ТУ (FK → ТочкаУчёта, CASCADE), НомерЗаявки (NOT NULL), ДатаНачала (NOT NULL), ДатаОкончания.
9. **Таблица `Примечание`** — 4 столбца: ИД_Примечания (PK), ИД_ТУ (FK → ТочкаУчёта, CASCADE), Текст (NOT NULL), ДатаСоздания (DEFAULT CURRENT_TIMESTAMP).

**Сводная схема внешних ключей.** В логической модели определено 8 внешних ключей, обеспечивающих ссылочную целостность между таблицами:

| Внешний ключ | Таблица-источник | Таблица-назначение | Правило удаления |
|:-------------|:-----------------|:-------------------|:-----------------|
| ИД_Организации | СтруктурноеПодразделение | Организация | CASCADE |
| ИД_Подразделения | КонтактноеЛицо | СтруктурноеПодразделение | CASCADE |
| ИД_Подразделения | Объект | СтруктурноеПодразделение | RESTRICT |
| ИД_МестаРасположения | Объект | МестоРасположения | RESTRICT |
| ИД_Объекта | ТочкаУчёта | Объект | CASCADE |
| ИД_Статуса | ТочкаУчёта | СтатусТУ | RESTRICT |
| ИД_ТУ | Заявка | ТочкаУчёта | CASCADE |
| ИД_ТУ | Примечание | ТочкаУчёта | CASCADE |

**Индексное обеспечение.** Для обеспечения производительности типовых запросов спроектированы индексы по следующим столбцам (Приложение Е, раздел 4):

а) индексы по внешним ключам — для ускорения операций соединения таблиц (JOIN) при выполнении запросов, объединяющих данные из связанных таблиц (например, поиск ТУ по объекту, заявок по ТУ);
б) уникальные индексы по бизнес-ключам — для обеспечения уникальности и быстрого поиска по номеру присоединения (ТочкаУчёта.НомерПрисоединения), номеру объекта в БД АльфаЦЕНТР (Объект.НомерВБДАльфаЦЕНТР) и наименованию статуса (СтатусТУ.Наименование).

**Домены атрибутов.** Для каждого атрибута определён домен — множество допустимых значений (Приложение Е, раздел 6). Ключевые домены включают:
а) целые положительные числа для всех идентификаторов (INTEGER, ≥ 1);
б) строки длиной до 255 символов для наименований;
в) строки длиной до 50 символов для номеров присоединения и номеров в БД АльфаЦЕНТР;
г) строки длиной до 100 символов для фамилии, имени, отчества;
д) логические значения для признака потребителя;
е) даты для дат начала и окончания заявок;
ж) метки времени для даты создания примечаний.

**Соответствие логической модели требованиям технического задания.** Разработанная логическая модель полностью соответствует требованиям, определённым в техническом задании (Приложение Д):

а) состав таблиц и столбцов соответствует требованиям раздела 3.1.1 ТЗ (9 сущностей с полным набором атрибутов);
б) первичные ключи реализованы как суррогатные целочисленные идентификаторы с автоматической генерацией (требование раздела 3.1.2 ТЗ);
в) внешние ключи и правила каскадного удаления соответствуют требованиям раздела 3.1.3 ТЗ;
г) ограничения целостности (NOT NULL, UNIQUE, DEFAULT, CHECK) соответствуют требованиям раздела 3.1.4 ТЗ;
д) индексы спроектированы в соответствии с требованиями раздела 3.2.2 ТЗ.
