## 3.2 Определение связей и ограничений (первичные и внешние ключи)

**Первичные ключи**

Для каждой из 9 таблиц справочной БД определён суррогатный первичный ключ с автоматической генерацией значений через `GENERATED ALWAYS AS IDENTITY`. Данное решение обеспечивает:

а) **независимость от бизнес-логики** — значения ключей не зависят от изменений атрибутов сущностей (например, номер присоединения ТУ может измениться, а идентификатор останется прежним);
б) **максимальную производительность** — целочисленные ключи обеспечивают эффективное построение индексов и соединение таблиц;
в) **единообразие именования** — все первичные ключи именуются по шаблону `ИД_<Сущность>`, что упрощает написание и сопровождение SQL-запросов.

**Перечень первичных ключей:**

| Таблица | Первичный ключ | Тип | Генерация |
|:--------|:---------------|:----|:----------|
| Организация | `ИД_Организации` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| СтруктурноеПодразделение | `ИД_Подразделения` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| КонтактноеЛицо | `ИД_КонтактногоЛица` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| МестоРасположения | `ИД_МестаРасположения` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| Объект | `ИД_Объекта` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| СтатусТУ | `ИД_Статуса` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| ТочкаУчёта | `ИД_ТУ` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| Заявка | `ИД_Заявки` | INTEGER | GENERATED ALWAYS AS IDENTITY |
| Примечание | `ИД_Примечания` | INTEGER | GENERATED ALWAYS AS IDENTITY |

**Внешние ключи и правила ссылочной целостности**

Между таблицами определены 8 внешних ключей, реализующих связи, установленные на этапе концептуального и логического проектирования (Приложение Г «ER-диаграмма», Приложение Е «Логическая модель базы данных»). Для каждого внешнего ключа явно задано правило каскадного удаления, соответствующее бизнес-семантике связи.

**Сводная таблица внешних ключей:**

| № | Внешний ключ | Таблица-источник | Таблица-назначение | Столбцы | Правило удаления |
|:-:|:-------------|:-----------------|:-------------------|:--------|:-----------------|
| 1 | FK_Подразделение_Организация | СтруктурноеПодразделение | Организация | ИД_Организации | CASCADE |
| 2 | FK_КонтактноеЛицо_Подразделение | КонтактноеЛицо | СтруктурноеПодразделение | ИД_Подразделения | CASCADE |
| 3 | FK_Объект_МестоРасположения | Объект | МестоРасположения | ИД_МестаРасположения | RESTRICT |
| 4 | FK_Объект_Подразделение | Объект | СтруктурноеПодразделение | ИД_Подразделения | RESTRICT |
| 5 | FK_ТочкаУчёта_Объект | ТочкаУчёта | Объект | ИД_Объекта | CASCADE |
| 6 | FK_ТочкаУчёта_СтатусТУ | ТочкаУчёта | СтатусТУ | ИД_Статуса | RESTRICT |
| 7 | FK_Заявка_ТочкаУчёта | Заявка | ТочкаУчёта | ИД_ТУ | CASCADE |
| 8 | FK_Примечание_ТочкаУчёта | Примечание | ТочкаУчёта | ИД_ТУ | CASCADE |

**Обоснование выбора правил каскадного удаления:**

1. **CASCADE (каскадное удаление)** применяется для связей, где дочерняя сущность не имеет смысла без родительской:
   - `Организация → СтруктурноеПодразделение` — при ликвидации организации её подразделения теряют юридическую основу;
   - `СтруктурноеПодразделение → КонтактноеЛицо` — контактные лица являются работниками подразделения;
   - `Объект → ТочкаУчёта` — точки учёта существуют только в пределах объекта;
   - `ТочкаУчёта → Заявка` и `ТочкаУчёта → Примечание` — заявки и примечания привязаны к конкретной ТУ.

2. **RESTRICT (запрет удаления)** применяется для связей, где удаление родительской записи недопустимо при наличии дочерних:
   - `МестоРасположения → Объект` — нельзя удалить место расположения, если в нём есть объекты (требуется предварительное переназначение);
   - `СтруктурноеПодразделение → Объект` — нельзя удалить подразделение, за которым закреплены объекты;
   - `СтатусТУ → ТочкаУчёта` — нельзя удалить статус, если он присвоен хотя бы одной ТУ.

Выбор RESTRICT вместо CASCADE для защищаемых родительских записей предотвращает случайную потерю данных и соответствует бизнес-требованию: удаление справочных значений (статусов, мест расположения, подразделений) должно быть возможно только после переназначения или удаления всех зависимых записей.

**Ограничения NOT NULL**

Все обязательные атрибуты, определённые в логической модели (Приложение Е), объявлены с ограничением NOT NULL на уровне DDL. Всего в схеме определено 30 обязательных полей (NOT NULL) из 39 атрибутов.

**Перечень необязательных (NULL) полей:**

| Таблица | Поле | Обоснование |
|:--------|:-----|:------------|
| КонтактноеЛицо | Имя, Отчество | Не у всех контактных лиц могут быть заполнены все части ФИО |
| КонтактноеЛицо | Должность | Должность может быть не указана для внешних контактных лиц |
| КонтактноеЛицо | Телефон | Не у всех контактных лиц есть служебный телефон |
| КонтактноеЛицо | Email | Электронная почта может отсутствовать |
| Организация | ПризнакПотребителя | Имеет значение DEFAULT FALSE, необязательность явного указания |
| Заявка | ДатаОкончания | Заявка может быть ещё не закрыта |
| Примечание | ДатаСоздания | Имеет значение DEFAULT CURRENT_TIMESTAMP |

**Ограничения UNIQUE**

Для атрибутов, уникальных по бизнес-логике, созданы три ограничения UNIQUE:

| Таблица | Столбец | Бизнес-смысл |
|:--------|:--------|:-------------|
| Объект | `НомерВБДАльфаЦЕНТР` | Каждый объект имеет уникальный номер в БД АльфаЦЕНТР |
| ТочкаУчёта | `НомерПрисоединения` | Каждая ТУ имеет уникальный номер присоединения/фидера |
| СтатусТУ | `Наименование` | Наименования статусов не должны дублироваться |

Ограничения UNIQUE гарантируют отсутствие дублирования данных на уровне СУБД и одновременно служат основой для создания уникальных индексов, ускоряющих поиск по данным атрибутам.

**Ограничение CHECK**

Для таблицы `СтатусТУ` установлено ограничение CHECK:

```sql
CONSTRAINT chk_статус_максвремя_положительное
    CHECK (МаксВремяОтсутствияДанныхЧас > 0)
```

Данное ограничение гарантирует, что максимально допустимое время отсутствия поступления данных в БД АльфаЦЕНТР является положительным числом. Нулевое или отрицательное значение не имеет физического смысла и приведёт к некорректной работе алгоритмов контроля своевременности данных в ПО AlphaMeterQC.

**Значения по умолчанию (DEFAULT)**

Для двух атрибутов заданы значения по умолчанию:

1. **`Организация.ПризнакПотребителя` = FALSE** — по умолчанию организация не является потребителем электроэнергии. Флаг устанавливается в TRUE только для организаций, которые непосредственно потребляют электроэнергию (например, ООО «Новоуренгойская энергосбытовая компания»).

2. **`Примечание.ДатаСоздания` = CURRENT_TIMESTAMP** — при создании примечания автоматически фиксируется текущие дата и время. Это исключает необходимость явной передачи значения из прикладного ПО и гарантирует точность временной метки.
