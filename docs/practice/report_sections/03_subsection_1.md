## 3.1 Создание базы данных и таблиц в выбранной СУБД

**Выбор и обоснование СУБД**

На этапе проектирования (раздел 2) в качестве целевой СУБД была определена PostgreSQL. Выбор обоснован следующими факторами:

а) **Открытый исходный код** — отсутствие лицензионных отчислений, соответствие политике импортозамещения, снижение совокупной стоимости владения;
б) **Поддержка стандарта SQL:2016** — совместимость с ПО AlphaMeterQC, возможность выполнения сложных аналитических запросов;
в) **Встроенные механизмы ссылочной целостности** — внешние ключи с каскадными операциями (CASCADE, RESTRICT), ограничения CHECK, уникальные ограничения;
г) **Ролевая модель разграничения доступа** — поддержка GRANT/REVOKE для реализации требований безопасности (раздел 5);
д) **Многопользовательский режим** — поддержка одновременного подключения нескольких АРМ инженеров;
е) **Надёжность и производительность** — механизмы транзакций (ACID), точки восстановления (SAVEPOINT), эффективная работа с индексами.

**Создание базы данных**

Создание базы данных выполняется от имени суперпользователя PostgreSQL следующей командой:

```sql
CREATE DATABASE alphameterqc
    WITH ENCODING 'UTF8'
         LC_COLLATE 'ru_RU.UTF-8'
         LC_CTYPE 'ru_RU.UTF-8'
         TEMPLATE template0;
```

Для логической группировки объектов базы данных создана схема `справочная_бд`, в которой размещены все таблицы, индексы и ограничения. Использование отдельной схемы позволяет:

а) изолировать объекты справочной БД от системных объектов PostgreSQL;
б) упростить управление правами доступа (роли назначаются на схему);
в) обеспечить возможность параллельного размещения нескольких схем в одной базе данных при необходимости расширения.

**Создание таблиц**

На основе логической модели (Приложение Е «Логическая модель базы данных») в СУБД PostgreSQL созданы 9 таблиц. Полный DDL-скрипт создания схемы приведён в Приложении И «DDL-скрипт создания схемы БД».

**Перечень созданных таблиц:**

| № | Таблица | Назначение | Количество атрибутов |
|:-:|:--------|:-----------|:--------------------:|
| 1 | `Организация` | Юридические лица — организации и потребители | 3 |
| 2 | `СтруктурноеПодразделение` | Структурные подразделения организаций | 3 |
| 3 | `КонтактноеЛицо` | Контактные лица структурных подразделений | 7 |
| 4 | `МестоРасположения` | Географические места расположения объектов | 2 |
| 5 | `Объект` | Объекты/электроустановки | 5 |
| 6 | `СтатусТУ` | Справочник статусов точек учёта | 3 |
| 7 | `ТочкаУчёта` | Точки учёта электрической энергии | 5 |
| 8 | `Заявка` | Заявки на выполнение работ по ТУ | 5 |
| 9 | `Примечание` | Текстовые примечания по ТУ | 4 |

**Ключевые проектные решения при создании таблиц:**

1. **Суррогатные первичные ключи.** Для каждой таблицы определён суррогатный первичный ключ с автоматической генерацией значений через `GENERATED ALWAYS AS IDENTITY`. Данное решение принято по следующим причинам:
   а) естественные ключи (например, номер присоединения для ТУ) могут изменяться в процессе эксплуатации;
   б) целочисленные ключи обеспечивают максимальную производительность при соединении таблиц и построении индексов;
   в) единообразие именования (`ИД_<Сущность>`) упрощает написание и сопровождение SQL-запросов.

2. **Явное указание ограничений NOT NULL.** Все обязательные атрибуты, определённые в логической модели, объявлены с ограничением NOT NULL на уровне DDL. Это гарантирует, что бизнес-требования к обязательности полей соблюдаются на уровне СУБД, независимо от прикладного ПО.

3. **Ограничения UNIQUE для бизнес-ключей.** Для атрибутов, уникальных по бизнес-логике, созданы ограничения UNIQUE:
   - `Объект.НомерВБДАльфаЦЕНТР` — уникальный номер объекта в БД АльфаЦЕНТР;
   - `ТочкаУчёта.НомерПрисоединения` — уникальный номер присоединения/фидера;
   - `СтатусТУ.Наименование` — уникальное наименование статуса.

4. **Ограничение CHECK для статусов ТУ.** Для поля `МаксВремяОтсутствияДанныхЧас` в таблице `СтатусТУ` установлено ограничение `CHECK (> 0)`, исключающее возможность указания нулевого или отрицательного значения.

5. **Значения по умолчанию.** Для полей `ПризнакПотребителя` (FALSE) и `ДатаСоздания` (CURRENT_TIMESTAMP) заданы значения по умолчанию, что сокращает объём вводимых данных при добавлении записей.

6. **Комментарии.** Для каждой таблицы и каждого столбца добавлены комментарии (COMMENT ON), описывающие назначение на русском языке. Комментарии облегчают сопровождение схемы и служат встроенной документацией базы данных.

**Создание индексов**

Для обеспечения производительности запросов в соответствии с требованиями раздела 3.2.2 технического задания (Приложение Д) созданы 10 индексов:

| № | Таблица | Столбец | Тип индекса | Назначение |
|:-:|:--------|:--------|:------------|:-----------|
| 1 | СтруктурноеПодразделение | ИД_Организации | B-tree | Ускорение поиска подразделений по организации |
| 2 | КонтактноеЛицо | ИД_Подразделения | B-tree | Ускорение поиска контактных лиц по подразделению |
| 3 | Объект | ИД_МестаРасположения | B-tree | Ускорение поиска объектов по месту расположения |
| 4 | Объект | ИД_Подразделения | B-tree | Ускорение поиска объектов по подразделению |
| 5 | Объект | НомерВБДАльфаЦЕНТР | B-tree (UNIQUE) | Уникальность и быстрый поиск по номеру в АльфаЦЕНТР |
| 6 | ТочкаУчёта | ИД_Объекта | B-tree | Ускорение поиска ТУ по объекту |
| 7 | ТочкаУчёта | ИД_Статуса | B-tree | Ускорение поиска ТУ по статусу |
| 8 | ТочкаУчёта | НомерПрисоединения | B-tree (UNIQUE) | Уникальность и быстрый поиск по номеру присоединения |
| 9 | Заявка | ИД_ТУ | B-tree | Ускорение поиска заявок по ТУ |
| 10 | Примечание | ИД_ТУ | B-tree | Ускорение поиска примечаний по ТУ |

Все индексы имеют тип B-tree, что является оптимальным выбором для операций точного поиска по значению (`WHERE ИД_Объекта = ?`) и соединения таблиц по внешним ключам. Индексы на столбцах внешних ключей созданы для всех дочерних таблиц, что обеспечивает эффективное выполнение запросов с соединением родительских и дочерних записей.

**Начальное наполнение справочников**

После создания схемы выполнено начальное наполнение справочных таблиц демонстрационными данными (Приложение К «DML-скрипт начального наполнения»). Наполнение включает:

а) **6 статусов ТУ** — ОРЭМ (2 часа), МТП (4 часа), ДСД (8 часов), НТ (24 часа), АСКУЭ (1 час), Резерв (48 часов);
б) **10 мест расположения** — адреса объектов в г. Новый Уренгой;
в) **10 организаций** — с признаком потребителя (TRUE/FALSE);
г) **10 структурных подразделений** — с привязкой к организациям;
д) **10 контактных лиц** — с должностями, телефонами и электронной почтой;
е) **10 объектов** — с уникальными номерами в БД АльфаЦЕНТР;
ж) **10 точек учёта** — с уникальными номерами присоединений;
з) **10 заявок** — с датами начала и окончания;
и) **10 примечаний** — с текстовыми описаниями.

Начальное наполнение позволяет проверить работоспособность схемы, корректность ссылочной целостности и выполнить тестовые запросы на реальных данных (раздел 4).
