# 6. Сбор исходной информации для выполнения выпускной квалификационной работы

## 6.1 Цель и задачи

В рамках этапа сбора исходной информации для выполнения выпускной квалификационной работы (ВКР) решалась задача систематизации материалов, полученных в ходе производственной практики, которые могут быть использованы в качестве основы для дальнейшей разработки темы ВКР.

## 6.2 Контекст решаемой задачи

В ходе производственной практики выполнялась работа по проектированию и разработке модуля справочной базы данных для программного средства «AlphaMeterQC», предназначенного для анализа и контроля своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии и хранящихся в базе данных Oracle программного комплекса «АльфаЦЕНТР».

Контекст решаемой задачи (Приложение А «Концепция создания продукта») определяет, что в рамках разработки ПО AlphaMeterQC планируется к разработке модуль экранных форм для вывода актуальной информации о своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии. Для формирования данных форм помимо информации из БД Oracle ПО АльфаЦЕНТР необходимо наличие дополнительной справочной информации, для хранения которой была разработана справочная база данных.

## 6.3 Собранные материалы для ВКР

В ходе производственной практики были собраны и систематизированы следующие материалы, которые могут быть использованы при выполнении ВКР:

**Аналитические материалы**

а) **Описание предметной области** — структура и бизнес-процессы учёта электрической энергии, роли участников (инженеры отдела системного сопровождения, руководитель отдела, команда разработки, администраторы, отдел информационной безопасности), описание текущего (AS-IS) и целевого (TO-BE) бизнес-процессов (Приложение В «Анализ бизнес-процессов предметной области»).

б) **Требования заинтересованных сторон** — 25 требований по 6 категориям (состав и структура данных, операции управления данными, интеграция, производительность, развёртывание, информационная безопасность), собранные методом интервьюирования пяти групп стейкхолдеров (Приложение Б «Требования заинтересованных сторон»).

в) **Техническое задание на проектирование БД** — документ, разработанный в соответствии с ГОСТ 34.602-2020, включающий требования к составу данных, производительности, API, безопасности, надёжности и развёртыванию (Приложение Д «Техническое задание на проектирование БД»).

**Проектные материалы**

г) **Концептуальная модель данных** — ER-диаграмма, описывающая 9 сущностей предметной области, их атрибуты, связи и правила каскадного удаления (Приложение Г «ER-диаграмма (модель «сущность-связь»)»).

д) **Логическая модель базы данных** — спецификация 9 таблиц с типами данных, ограничениями, первичными и внешними ключами, индексами, доменами атрибутов, результатами нормализации до 3НФ (Приложения Е и Ж).

е) **Физическая реализация** — DDL-скрипт создания схемы БД в PostgreSQL 15+, DML-скрипт начального наполнения, DCL-скрипт настройки безопасности (Приложения И, К, Л).

**Результаты тестирования и оптимизации**

ж) **Оптимизация запросов** — 8 типовых запросов с планами выполнения (EXPLAIN ANALYZE), анализ производительности, рекомендации по масштабированию (Приложение Н).

з) **Нагрузочное тестирование** — 10 сценариев (SELECT, INSERT, UPDATE, DELETE) при объёме данных до 10 000 ТУ и 30 000 заявок, подтверждение соответствия требованиям производительности с запасом от 460× до 3 800× (Приложения О, П).

и) **Анализ узких мест** — 6 потенциальных узких мест с прогнозами при росте объёмов и рекомендациями по устранению (Приложение Р).

**Эксплуатационная документация**

к) **Инструкция по развёртыванию и эксплуатации** — порядок установки PostgreSQL, выполнения скриптов, настройки подключения, резервного копирования, мониторинга (Приложение М).

л) **Скрипты резервного копирования и восстановления** — bash-скрипты для pg_dump/pg_restore с автоматической ротацией (Приложения С, Т).

м) **Политика безопасности** — ролевая модель (3 роли), аудит изменений (9 триггеров), защита соединений (TLS/SSL), процедуры реагирования на инциденты (Приложение У).

## 6.4 Направления дальнейшей разработки темы ВКР

На основе материалов, собранных в ходе производственной практики, могут быть определены следующие направления для выполнения ВКР:

1. **Разработка модуля экранных форм ПО AlphaMeterQC** — непосредственное продолжение выполненной работы: разработка пользовательского интерфейса для отображения и редактирования справочных данных, интеграция справочной БД с основным приложением.

2. **Разработка подсистемы анализа и мониторинга данных со счётчиков электрической энергии** — создание аналитических отчётов и дашбордов на основе данных, комбинируемых из справочной БД и БД Oracle АльфаЦЕНТР.

3. **Разработка модуля автоматического контроля своевременности поступления данных** — реализация алгоритмов проверки соответствия фактического времени поступления данных нормативным значениям (МаксВремяОтсутствияДанныхЧас) с формированием оповещений.

4. **Разработка веб-интерфейса для управления справочной БД** — создание кросс-платформенного интерфейса для удалённого администрирования справочных данных.
