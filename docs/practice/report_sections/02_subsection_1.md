## 2.1 Создание ER-диаграммы (модель «сущность-связь»)

На основе результатов анализа требований заинтересованных сторон (Приложение Б) и определённых ключевых сущностей и атрибутов данных (пункт 1.2) была разработана концептуальная модель данных в форме ER-диаграммы (модель «сущность-связь»). Данная диаграмма является графическим представлением структуры данных справочной базы данных модуля AlphaMeterQC и служит основой для последующего проектирования логической модели (пункт 2.2).

**Методология и нотация.** Для построения ER-диаграммы была выбрана нотация «Crow's Foot» (IBM Information Engineering), которая обеспечивает наглядное представление типов связей (1:1, 1:M, M:M) и правил обязательности участия сущностей в связях. Данная нотация широко применяется при проектировании реляционных баз данных и поддерживается большинством CASE-средств. В качестве инструмента моделирования использован PlantUML — текстовый язык описания диаграмм, интегрированный с системой контроля версий Git, что обеспечивает версионирование модели наряду с остальными артефактами проекта.

**Процесс создания ER-диаграммы.** Разработка концептуальной модели выполнялась в следующей последовательности:

1) **Определение сущностей.** На основе анализа предметной области (пункт 1.1) и результатов выявления ключевых сущностей (пункт 1.2) были зафиксированы 9 сущностей: Организация, СтруктурноеПодразделение, КонтактноеЛицо, МестоРасположения, Объект, ТочкаУчёта, СтатусТУ, Заявка и Примечание. Каждая сущность представляет собой именованный класс объектов предметной области, обладающий одинаковым набором атрибутов.

2) **Определение атрибутов.** Для каждой сущности были определены атрибуты с указанием типа данных и ограничений:
   а) первичные ключи (PK) — уникальные идентификаторы записей (тип INTEGER);
   б) внешние ключи (FK) — ссылки на связанные сущности (тип INTEGER);
   в) обязательные атрибуты (NOT NULL) — атрибуты, которые не могут принимать неопределённое значение (например, Наименование организации, НомерПрисоединения ТУ);
   г) опциональные атрибуты — атрибуты, допускающие NULL (например, ДатаОкончания заявки, Отчество контактного лица).

3) **Определение связей.** Между сущностями были установлены бинарные связи типа «один ко многим» (1:M), отражающие реальные бизнес-правила предметной области. Для каждой связи определены:
   а) родительская (главная) и дочерняя (зависимая) сущности;
   б) правило каскадного удаления (CASCADE или RESTRICT) в зависимости от семантики связи.

4) **Построение графической схемы.** С использованием языка PlantUML была построена графическая схема, на которой сущности представлены в виде таблиц с перечнем атрибутов, а связи — в виде линий с обозначением типа связи и краткой подписью (например, «имеет», «содержит», «классифицирует»).

**Описание полученной ER-диаграммы.** Полная ER-диаграмма с детальным отображением всех сущностей, атрибутов, связей и правил каскадного удаления приведена в Приложении Г «ER-диаграмма (модель «сущность-связь»)» (Приложение Г, раздел 1). Ниже приведено краткое описание ключевых элементов модели.

*Сущности и их назначение:*

а) **Организация** — юридическое лицо, которое может являться Потребителем электрической энергии. Содержит атрибуты: ИД_Организации (PK), Наименование, ПризнакПотребителя.
б) **СтруктурноеПодразделение** — подразделение организации, в эксплуатационной ответственности которого находятся объекты. Содержит атрибуты: ИД_Подразделения (PK), ИД_Организации (FK), Наименование.
в) **КонтактноеЛицо** — работник подразделения с контактными данными. Содержит атрибуты: ИД_КонтактногоЛица (PK), ИД_Подразделения (FK), Фамилия, Имя, Отчество, Должность, Телефон, Email.
г) **МестоРасположения** — географическое место расположения объектов. Содержит атрибуты: ИД_МестаРасположения (PK), Наименование.
д) **Объект** — объект/электроустановка, в пределах которого осуществляется учёт электроэнергии. Содержит атрибуты: ИД_Объекта (PK), Наименование, НомерВБДАльфаЦЕНТР, ИД_МестаРасположения (FK), ИД_Подразделения (FK).
е) **ТочкаУчёта** — центральная сущность модели, точка учёта электрической энергии. Содержит атрибуты: ИД_ТУ (PK), Наименование, НомерПрисоединения, ИД_Объекта (FK), ИД_Статуса (FK).
ж) **СтатусТУ** — справочник статусов ТУ с нормативным временем отсутствия данных. Содержит атрибуты: ИД_Статуса (PK), Наименование, МаксВремяОтсутствияДанныхЧас.
з) **Заявка** — заявка на выполнение работ по ТУ. Содержит атрибуты: ИД_Заявки (PK), ИД_ТУ (FK), НомерЗаявки, ДатаНачала, ДатаОкончания.
и) **Примечание** — текстовое примечание по ТУ. Содержит атрибуты: ИД_Примечания (PK), ИД_ТУ (FK), Текст, ДатаСоздания.

*Связи между сущностями:*

| Связь | Тип | Родительская сущность | Дочерняя сущность | Правило удаления |
|:------|:---:|:----------------------|:-------------------|:-----------------|
| Организация — Подразделение | 1:M | Организация | СтруктурноеПодразделение | CASCADE |
| Подразделение — КонтактноеЛицо | 1:M | СтруктурноеПодразделение | КонтактноеЛицо | CASCADE |
| Подразделение — Объект | 1:M | СтруктурноеПодразделение | Объект | RESTRICT |
| МестоРасположения — Объект | 1:M | МестоРасположения | Объект | RESTRICT |
| Объект — ТочкаУчёта | 1:M | Объект | ТочкаУчёта | CASCADE |
| СтатусТУ — ТочкаУчёта | 1:M | СтатусТУ | ТочкаУчёта | RESTRICT |
| ТочкаУчёта — Заявка | 1:M | ТочкаУчёта | Заявка | CASCADE |
| ТочкаУчёта — Примечание | 1:M | ТочкаУчёта | Примечание | CASCADE |

**Обоснование выбора правил каскадного удаления.** Правила каскадного удаления выбраны исходя из семантики бизнес-связей:

а) **CASCADE** применяется для связей, где дочерняя сущность не имеет самостоятельного смысла без родительской: при удалении организации удаляются её подразделения; при удалении подразделения — его контактные лица; при удалении объекта — его точки учёта; при удалении точки учёта — её заявки и примечания. Это обеспечивает ссылочную целостность и предотвращает появление «осиротевших» записей.
б) **RESTRICT** применяется для связей, где удаление родительской записи не должно происходить, если существуют связанные дочерние записи: запрещено удаление подразделения, за которым закреплены объекты; запрещено удаление места расположения, в котором есть объекты; запрещено удаление статуса, который присвоен точкам учёта. Это предотвращает случайную потерю критически важных данных.

**Связь ER-диаграммы с предшествующими артефактами.** Разработанная ER-диаграмма является графической реализацией результатов предыдущих этапов:
а) состав и атрибуты сущностей соответствуют результатам определения ключевых сущностей (пункт 1.2);
б) бизнес-правила, отражённые в связях, основаны на анализе бизнес-процессов (Приложение В);
в) требования к целостности данных (каскадные операции) соответствуют спецификации технического задания (Приложение Д, раздел 3.1.3).
