## 4.3 Анализ и устранение узких мест в производительности

**Цель и методика анализа**

На этапе анализа узких мест решалась задача выявления потенциальных проблем производительности справочной БД AlphaMeterQC, которые могут проявиться при росте объёма данных или изменении характера нагрузки. Анализ проводился на основе результатов нагрузочного тестирования (Приложение О) и оптимизации запросов (Приложение Н).

Методика анализа включала:

1. **Сбор данных о производительности** — на основе 10 сценариев нагрузочного тестирования и 8 типовых запросов.
2. **Идентификацию потенциальных узких мест** — выявление запросов с наибольшим временем выполнения, операций полного сканирования таблиц (Seq Scan), неоптимальных планов выполнения.
3. **Оценку влияния** — определение критичности каждого узкого места для эксплуатации БД при текущих и прогнозируемых объёмах данных.
4. **Разработку рекомендаций** — предложения по устранению или смягчению выявленных узких мест.

**Идентифицированные узкие места**

На основе анализа планов выполнения и прогнозирования поведения при росте объёма данных были идентифицированы 6 потенциальных узких мест. Полный анализ каждого узкого места с планами выполнения, прогнозами и рекомендациями приведён в Приложении Р «Анализ и устранение узких мест в производительности».

**Узкое место №1: Полное сканирование таблицы ТочкаУчёта в агрегирующем запросе (T4).**

Запрос статистики ТУ по статусам выполняет полное сканирование (Seq Scan) таблицы `ТочкаУчёта`. При 10 000 записей время составляет 2,168 мс, что значительно ниже порога в 1 000 мс. Однако при росте объёма до 1 000 000 записей прогнозируемое время составит ~220 мс.

*Рекомендация:* создать материализованное представление `мв_статистика_ту_по_статусам` с предрассчитанной статистикой и ежедневным обновлением. Это позволит выполнять запрос за < 0,1 мс независимо от объёма данных.

**Узкое место №2: Отсутствие композитного индекса для иерархического запроса (T2).**

Запрос полной иерархии соединяет 5 таблиц последовательными вложенными циклами. При текущем объёме (10 000 ТУ) время — 0,112 мс. При росте до 100 000 ТУ прогнозируется ~1–2 мс.

*Рекомендация:* добавить композитный индекс `idx_ту_объект_статус(ИД_Объекта, ИД_Статуса)` для покрытия соединения ТочкаУчёта → СтатусТУ, что позволит выполнять Index Only Scan.

**Узкое место №3: Отсутствие индекса для фильтрации заявок по дате (T5).**

Запрос заявок по ТУ с фильтром по дате использует Index Scan по `idx_заявка_ту` с последующей фильтрацией (Filter). При 30 000 заявок время — 0,022 мс.

*Рекомендация:* добавить композитный индекс `idx_заявка_ту_дата(ИД_ТУ, ДатаОкончания)` для покрытия обоих условий без дополнительной фильтрации.

**Узкое место №4: Каскадное удаление при массовых операциях (T10).**

При удалении ТУ выполняются каскадные операции через триггеры внешних ключей. При единичном удалении время — 0,312 мс. При массовом удалении (например, удаление организации с 500 ТУ) прогнозируется ~150 мс.

*Рекомендация:* при необходимости массового удаления выполнять операцию в транзакции с предварительным удалением дочерних записей (пакетная обработка).

**Узкое место №5: Разрастание мёртвых кортежей при интенсивных DML-операциях.**

Механизм MVCC PostgreSQL приводит к накоплению мёртвых кортежей при UPDATE и DELETE. При интенсивных операциях (частые обновления статусов ТУ) количество мёртвых кортежей может расти быстрее, чем autovacuum успевает обрабатывать.

*Рекомендация:* настроить autovacuum для таблиц с высокой интенсивностью DML (ТочкаУчёта, Заявка) с уменьшенными порогами срабатывания; организовать мониторинг мёртвых кортежей.

**Узкое место №6: Отсутствие мониторинга производительности.**

В текущей конфигурации не настроен мониторинг производительности БД. При росте объёма данных или изменении характера нагрузки ухудшение производительности может быть обнаружено только по жалобам пользователей.

*Рекомендация:* внедрить базовый мониторинг с использованием системных представлений PostgreSQL (`pg_stat_statements`, `pg_stat_user_tables`); настроить оповещения при превышении порогов времени выполнения запросов (> 500 мс), количества мёртвых кортежей (> 20 %) и свободного места на диске (< 20 %).
