## 5.2 Обеспечение безопасности данных: контроль доступа и шифрование

**Цель и задачи**

На этапе обеспечения безопасности данных решалась задача реализации требований технического задания (Приложение Д, раздел 3.4) к разграничению прав доступа, аудиту изменений и защите соединений справочной БД AlphaMeterQC.

**Ролевая модель разграничения доступа**

В соответствии с требованиями ТЗ (Приложение Д, раздел 3.4.1) были созданы три роли доступа (реализованы в DCL-скрипте, Приложение Л):

| Роль | Права | Назначение |
|:-----|:------|:-----------|
| `amqc_reader` | SELECT на все таблицы схемы `справочная_бд` | Все авторизованные пользователи ПО AlphaMeterQC |
| `amqc_writer` | SELECT, INSERT, UPDATE, DELETE на все таблицы; USAGE на последовательности | Уполномоченные сотрудники, ответственные за ведение справочных данных |
| `amqc_admin` | ALL PRIVILEGES на все объекты схемы | Администраторы базы данных |

Ролевая модель основана на принципе минимальных привилегий (POLP): пользователь получает только те права, которые необходимы для выполнения его должностных обязанностей. Роли чтения и записи разделены — пользователь, имеющий право на изменение данных, не может быть одновременно обычным читающим пользователем.

Для автоматического назначения прав на новые таблицы, создаваемые в будущем, использована конструкция `ALTER DEFAULT PRIVILEGES`, что исключает необходимость ручного назначения прав после каждого изменения схемы.

**Аудит изменений данных**

В соответствии с требованиями ТЗ (Приложение Д, раздел 3.4.2) был реализован журнал аудита изменений данных (Приложение Л, раздел 7).

**Таблица аудита** `ЖурналАудита` содержит следующие поля:
а) `ИД_Записи` — уникальный идентификатор записи аудита (BIGINT, первичный ключ);
б) `ИмяПользователя` — имя пользователя, выполнившего операцию (current_user);
в) `ВремяИзменения` — время выполнения операции (TIMESTAMP, по умолчанию CURRENT_TIMESTAMP);
г) `ТипОперации` — тип операции: INSERT, UPDATE, DELETE (с ограничением CHECK);
д) `ИмяТаблицы` — имя изменённой таблицы;
е) `ИД_ЗаписиТаблицы` — идентификатор изменённой записи;
ж) `СтарыеЗначения` — значения до изменения в формате JSONB (для UPDATE и DELETE);
з) `НовыеЗначения` — значения после изменения в формате JSONB (для INSERT и UPDATE).

**Триггеры аудита.** Для каждой из 9 таблиц создан триггер `AFTER INSERT OR UPDATE OR DELETE`, вызывающий функцию `фн_аудит_изменений()`. Функция в зависимости от типа операции формирует запись в журнале аудита: для INSERT фиксируются новые значения, для UPDATE — старые и новые, для DELETE — старые значения.

**Защита журнала аудита.** Права `DELETE` и `UPDATE` на таблицу `ЖурналАудита` отозваны у ролей `amqc_reader` и `amqc_writer`. Только роль `amqc_admin` имеет полный доступ к журналу. Для ускорения поиска созданы индексы по полям `ВремяИзменения`, `ИмяТаблицы` и `ИмяПользователя`.

**Защита соединений (TLS/SSL)**

В соответствии с требованиями ТЗ (Приложение Д, раздел 3.4.3) настроена защита соединений между справочной БД и АРМ инженеров по протоколу TLS/SSL (Приложение У, раздел 4).

**Настройка сертификатов.** Сгенерирован самоподписанный сертификат сервера, настроены параметры `ssl = on`, `ssl_cert_file` и `ssl_key_file` в конфигурации PostgreSQL.

**Настройка pg_hba.conf.** Для всех ролей настроен обязательный режим SSL (`hostssl`) с аутентификацией `scram-sha-256`:

```
hostssl    alphameterqc    amqc_reader     192.168.0.0/16    scram-sha-256
hostssl    alphameterqc    amqc_writer     192.168.0.0/16    scram-sha-256
hostssl    alphameterqc    amqc_admin      127.0.0.1/32      scram-sha-256
```

**Сетевая изоляция.** Сервер БД размещается в выделенном сегменте локальной сети. Доступ к порту 5432 разрешён только с АРМ инженеров (список разрешённых IP-адресов). Доступ по SSH — только для администраторов.
