## 4.2 Проведение нагрузочного тестирования

**Цель и методика нагрузочного тестирования**

На этапе нагрузочного тестирования решалась задача проверки соответствия справочной БД AlphaMeterQC требованиям технического задания (Приложение Д, раздел 3.2.1) к производительности при объёме данных до 10 000 записей. В ходе тестирования проверялись следующие критерии:

а) время выполнения типового запроса на чтение не превышает 1 секунды;
б) время вставки одной записи не превышает 0,5 секунды;
в) время обновления одной записи не превышает 0,5 секунды.

Методика тестирования включала:

1. **Подготовку тестовых данных** — генерацию данных в масштабах, приближенных к реальной эксплуатации (до 10 000 ТУ, 30 000 заявок и примечаний).
2. **Определение сценариев тестирования** — 10 сценариев, покрывающих все типы операций (SELECT, INSERT, UPDATE, DELETE) и характерные нагрузки (одиночные операции, массовые операции, каскадные удаления).
3. **Выполнение сценариев с EXPLAIN (ANALYZE, BUFFERS)** — получение плана выполнения, фактического времени и информации о буферном кэше.
4. **Фиксацию и анализ результатов** — сравнение с требованиями ТЗ, оценку эффективности индексов, выявление узких мест.

**Подготовка тестовых данных**

Для нагрузочного тестирования был разработан скрипт генерации тестовых данных (Приложение П «Скрипт генерации тестовых данных для нагрузочного тестирования»). Скрипт генерирует данные с использованием генератора последовательностей `generate_series()` и случайного распределения (`random()`), что позволяет имитировать реальную структуру данных.

**Объём сгенерированных тестовых данных:**

| Таблица | Количество записей | Характер данных |
|:--------|:------------------:|:----------------|
| Организация | 100 | Юридические лица с признаком потребителя (70 % — потребители) |
| СтруктурноеПодразделение | 500 | Подразделения, случайным образом привязанные к организациям |
| КонтактноеЛицо | 1 000 | Контактные лица со случайными должностями |
| МестоРасположения | 200 | Адреса мест расположения |
| Объект | 1 000 | Объекты с уникальными номерами в БД АльфаЦЕНТР |
| ТочкаУчёта | 10 000 | ТУ со случайными статусами, привязанные к объектам |
| СтатусТУ | 6 | Справочник статусов (без изменений) |
| Заявка | 30 000 | Заявки со случайными датами (60 % — закрытые) |
| Примечание | 30 000 | Примечания со случайными текстами |

Общий объём тестовых данных составил **~72 800 записей** в 9 таблицах, что превышает требуемый порог в 10 000 записей для таблицы ТочкаУчёта и позволяет оценить поведение системы при нагрузке, превышающей номинальную.

**Сценарии тестирования**

Было разработано 10 сценариев тестирования, охватывающих все типовые операции:

| № | Сценарий | Тип | Описание |
|:-:|:---------|:---|:---------|
| T1 | Список ТУ по объекту со статусом | SELECT | Выборка ТУ по ИД_Объекта с присоединением статуса |
| T2 | Полная иерархия (5 таблиц) | SELECT | Соединение Организация → Подразделение → Объект → ТУ → Статус с сортировкой |
| T3 | Поиск ТУ по номеру присоединения | SELECT | Точный поиск по уникальному индексу |
| T4 | Статистика ТУ по статусам | SELECT | GROUP BY с агрегацией COUNT (полное сканирование) |
| T5 | Заявки по ТУ с фильтром по дате | SELECT | Выборка заявок за период с условием по дате |
| T6 | Вставка одной ТУ | INSERT | Добавление одной записи в таблицу ТочкаУчёта |
| T7 | Массовая вставка 100 ТУ | INSERT | Вставка 100 записей в одной транзакции |
| T8 | Обновление статуса одной ТУ | UPDATE | Изменение ИД_Статуса для одной ТУ |

**Результаты тестирования**

Каждый сценарий выполнялся с использованием `EXPLAIN (ANALYZE, BUFFERS)` для получения плана выполнения, фактического времени и информации об использовании буферного кэша. Полные планы выполнения приведены в Приложении О «Результаты нагрузочного тестирования».

**Анализ результатов**

**Соответствие требованиям.** Все 10 сценариев нагрузочного тестирования успешно пройдены. Фактическое время выполнения значительно ниже установленных требований:

а) **Чтение (SELECT):** максимальное время — 2,168 мс (T4 — статистика по статусам) при требуемом пороге 1 000 мс. Запас прочности — **460×**.
б) **Вставка (INSERT):** максимальное время — 0,131 мс на одну запись при требуемом пороге 500 мс. Запас прочности — **3 800×**.
в) **Обновление (UPDATE):** максимальное время — 0,198 мс на одну запись при требуемом пороге 500 мс. Запас прочности — **2 500×**.
г) **Удаление (DELETE):** время 0,312 мс с учётом каскадного удаления зависимых записей при требуемом пороге 500 мс. Запас прочности — **1 600×**.

**Эффективность индексов.** Все запросы на чтение (кроме T4) используют индексный поиск (Index Scan). Индексы, созданные на этапе реализации (Приложение И), полностью покрывают потребности типовых запросов:

а) `idx_ту_объект` — обеспечивает быстрый поиск ТУ по объекту (T1);
б) `idx_ту_номер_присоединения` — обеспечивает быстрый поиск по номеру присоединения (T3);
в) `idx_заявка_ту` — обеспечивает быстрый поиск заявок по ТУ (T5);
г) `idx_объект_подразделение`, `idx_подразделение_организация` — обеспечивают эффективное соединение в иерархическом запросе (T2).

**Поведение при пиковой нагрузке.** Сценарий T4 (статистика ТУ по статусам) показал наибольшее время — 2,168 мс. Это связано с полным сканированием таблицы `ТочкаУчёта` (10 000 строк) для агрегации. Данное поведение является ожидаемым и корректным: для запроса, которому необходимо прочитать все строки таблицы, Seq Scan эффективнее Index Scan. При необходимости ускорения данного запроса может быть создан материализованный View с периодическим обновлением.

**Каскадные операции.** Сценарий T10 (удаление ТУ) подтвердил корректную работу каскадного удаления: триггеры внешних ключей `fk_заявка_ту` и `fk_примечание_ту` отработали за 0,018 мс и 0,015 мс соответственно. Ссылочная целостность поддерживается на уровне СУБД без потери производительности.
