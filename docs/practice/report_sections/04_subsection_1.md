## 4.1 Оптимизация запросов для повышения производительности

**Цель и методика оптимизации**

На этапе оптимизации запросов решалась задача обеспечения выполнения требований технического задания (Приложение Д, раздел 3.2.1) к производительности: время выполнения типового запроса на чтение не должно превышать 1 секунды при объёме данных до 10 000 записей, время вставки и обновления одной записи — не более 0,5 секунды.

Методика оптимизации включала последовательное выполнение следующих шагов:

1. **Определение типовых запросов** — на основе требований технического задания (Приложение Д, раздел 3.3.2) и бизнес-процессов предметной области (Приложение В) выделены 8 типовых запросов, покрывающих все сценарии использования справочной БД в ПО AlphaMeterQC.
2. **Выполнение запросов с EXPLAIN ANALYZE** — для каждого запроса получен план выполнения и фактическое время выполнения на демонстрационных данных (60 записей в 9 таблицах).
3. **Анализ плана выполнения** — выявление операций полного сканирования таблиц (Seq Scan), оценка стоимости операций, проверка использования индексов.
4. **Оптимизация** — добавление недостающих индексов, переписывание запросов, настройка параметров конфигурации СУБД.
5. **Повторное тестирование** — сравнение времени выполнения до и после оптимизации.

**Определение типовых запросов**

На основе требований технического задания (Приложение Д, раздел 3.3.2) и анализа бизнес-процессов (Приложение В) были выделены следующие типовые запросы, которые будут выполняться ПО AlphaMeterQC при формировании экранных форм:

| № | Запрос | Назначение | Количество соединяемых таблиц |
|:-:|:-------|:-----------|:----------------------------:|
| 1 | Список ТУ по объекту с информацией о статусе | Отображение всех точек учёта выбранного объекта | 2 |
| 2 | Список заявок по ТУ | Отображение заявок, связанных с выбранной ТУ | 1 |
| 3 | Список примечаний по ТУ | Отображение примечаний по выбранной ТУ | 1 |
| 4 | Полная иерархия «Организация → Подразделение → Объект → ТУ» | Отображение иерархической структуры по организации | 5 |
| 5 | Контактные лица подразделения по объекту | Отображение контактных данных ответственных лиц | 3 |
| 6 | Поиск ТУ по номеру присоединения | Быстрый поиск ТУ по номеру фидера в БД АльфаЦЕНТР | 3 |
| 7 | Объекты по месту расположения | Фильтрация объектов по месту расположения | 2 |
| 8 | Статистика ТУ в разрезе статусов | Аналитическая панель — распределение ТУ по статусам | 2 |

Полные тексты запросов, планы выполнения и результаты анализа приведены в Приложении Н «Оптимизация запросов справочной БД».

**Анализ планов выполнения запросов**

Для каждого из 8 типовых запросов был выполнен `EXPLAIN ANALYZE` — получен план выполнения и фактическое время выполнения на демонстрационных данных. Анализ планов выполнения показал следующие результаты.

**Запросы 1–3, 5–7 (простые запросы с соединением 1–3 таблиц).** Все запросы используют индексный поиск (Index Scan) по существующим индексам, созданным на этапе реализации (Приложение И). Планы выполнения показывают вложенные циклы (Nested Loop), что является оптимальным решением для малого объёма данных. Время выполнения составило от 0,018 до 0,036 мс.

**Запрос 4 (полная иерархия, 5 таблиц).** Наиболее сложный запрос, соединяющий таблицы `Организация`, `СтруктурноеПодразделение`, `Объект`, `ТочкаУчёта` и `СтатусТУ`. План выполнения показал каскад вложенных циклов (Nested Loop) с использованием индексов на всех этапах соединения:
а) поиск организации по первичному ключу (`pk_организация`);
б) поиск подразделений по индексу внешнего ключа (`idx_подразделение_организация`);
в) поиск объектов по индексу внешнего ключа (`idx_объект_подразделение`);
г) поиск ТУ по индексу внешнего ключа (`idx_ту_объект`);
д) поиск статуса по первичному ключу (`pk_статусту`).

Сортировка результата выполнялась в памяти (quicksort, 25 kB). Время выполнения составило 0,078 мс.

**Запрос 8 (статистика ТУ по статусам).** Для таблиц малого объёма (СтатусТУ — 6 строк, ТочкаУчёта — 10 строк) оптимизатор PostgreSQL выбрал полное сканирование (Seq Scan) малых таблиц, что является корректным решением: стоимость полного сканирования малой таблицы ниже стоимости обращения к индексу. Время выполнения составило 0,052 мс.

|:-:|:-------|:---------------------:|:--------------------:|:---------------------:|

**Вывод:** Все типовые запросы выполняются за время, значительно меньшее требуемого порога в 1 секунду. Максимальное измеренное время выполнения составило 0,078 мс (запрос 4 — полная иерархия), что более чем в 12 800 раз быстрее установленного требования. Существующие индексы полностью покрывают потребности типовых запросов. Дополнительная оптимизация на текущем этапе не требуется.

**Дополнительные оптимизации и рекомендации по масштабированию**

Несмотря на то, что текущая производительность удовлетворяет требованиям с большим запасом, в рамках работ по оптимизации были подготовлены рекомендации на случай роста объёма данных до 10 000 записей и более (Приложение Н, разделы 4 и 6).

**Композитный индекс для иерархического запроса.** Для запроса 4 (полная иерархия) при росте объёма данных рекомендуется добавить композитный индекс:

```sql
CREATE INDEX idx_ту_объект_статус
    ON справочная_бд.ТочкаУчёта(ИД_Объекта, ИД_Статуса);
```

Данный индекс является покрывающим для запроса 4 — все необходимые столбцы для соединения содержатся в самом индексе, что позволит выполнять Index Only Scan без обращения к таблице.

**Настройка параметров конфигурации PostgreSQL.** Для обеспечения стабильной производительности при росте объёма данных рекомендуется настроить следующие параметры:

а) `work_mem` = 4–8 MB — для выполнения сортировок (ORDER BY) в памяти, а не на диске;
б) `shared_buffers` = 25 % от ОЗУ сервера — для кэширования часто используемых таблиц и индексов;
в) `effective_cache_size` = 50–75 % от ОЗУ сервера — для оценки оптимизатором доступного кэша;
г) `random_page_cost` = 1.1 (для SSD) — для корректной оценки стоимости индексного доступа на SSD-накопителях;
д) `default_statistics_target` = 100–200 — для более точной статистики оптимизатора.

**Регулярное обслуживание.** Для поддержания производительности на постоянной основе рекомендуется:

1. **Обновление статистики** — выполнять `ANALYZE` после массовых загрузок данных (более 10 % записей).
2. **Дефрагментация индексов** — выполнять `REINDEX` при значительном объёме модификаций (более 20 % записей).
3. **Автовакуумирование** — настроить `autovacuum` для предотвращения разрастания мёртвых кортежей.
