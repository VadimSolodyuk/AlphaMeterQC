## 3.3 Разработка SQL-скриптов для создания и управления объектами базы данных

**Состав разработанных SQL-скриптов**

В рамках физической реализации базы данных разработан комплект SQL-скриптов, охватывающий все стадии жизненного цикла базы данных: создание схемы (DDL), начальное наполнение (DML), настройку безопасности (DCL) и документирование процесса развёртывания.

**Перечень разработанных скриптов:**

| № | Файл | Тип | Назначение |
|:-:|:-----|:---|:-----------|
| 1 | `08_ddl_schema.sql` | DDL | Создание схемы, 9 таблиц, первичных и внешних ключей, 10 индексов, ограничений UNIQUE/CHECK, значений DEFAULT, комментариев |
| 2 | `09_dml_seed.sql` | DML | Первичное наполнение справочников демонстрационными данными (60 записей) |
| 3 | `10_dcl_security.sql` | DCL | Создание ролей доступа, разграничение прав, таблица аудита, триггеры аудита |
| 4 | `11_deployment_guide.md` | Документация | Инструкция по развёртыванию и эксплуатации БД |

Полные тексты скриптов приведены в Приложениях И, К, Л, М соответственно.

**DDL-скрипт создания схемы (Приложение И)**

DDL-скрипт `08_ddl_schema.sql` является основным скриптом, определяющим структуру базы данных. Он включает:

**Структурные элементы:**

а) создание схемы `справочная_бд` для логической группировки объектов;
б) создание 9 таблиц с полным набором атрибутов, типов данных и ограничений;
в) определение 9 первичных ключей с автоматической генерацией (`GENERATED ALWAYS AS IDENTITY`);
г) определение 8 внешних ключей с явно заданными правилами каскадного удаления (4 CASCADE, 3 RESTRICT);
д) создание 10 индексов типа B-tree для обеспечения производительности запросов;
е) установку 3 ограничений UNIQUE для бизнес-ключей;
ж) установку 1 ограничения CHECK для поля `МаксВремяОтсутствияДанныхЧас > 0`;
з) задание 2 значений по умолчанию (DEFAULT);
и) добавление комментариев (COMMENT ON) для каждой таблицы и каждого столбца.

**Порядок создания таблиц** выбран с учётом зависимостей по внешним ключам: сначала создаются независимые таблицы (Организация, МестоРасположения, СтатусТУ), затем зависимые (СтруктурноеПодразделение, КонтактноеЛицо, Объект), и в последнюю очередь — таблицы, зависящие от всех остальных (ТочкаУчёта, Заявка, Примечание). Такой порядок гарантирует успешное выполнение скрипта без ошибок ссылочной целостности.

**Обработка ошибок.** Скрипт использует идемпотентные конструкции: `CREATE SCHEMA IF NOT EXISTS`, что позволяет выполнять его многократно без риска дублирования объектов. При повторном выполнении скрипт завершится с ошибкой на этапе создания уже существующих таблиц — это предусмотрено для предотвращения случайного пересоздания схемы.

**DML-скрипт начального наполнения (Приложение К)**

DML-скрипт `09_dml_seed.sql` предназначен для первичного наполнения справочных таблиц демонстрационными данными. Скрипт включает вставку 60 записей в 9 таблиц:

| Таблица | Количество записей | Характер данных |
|:--------|:------------------:|:----------------|
| СтатусТУ | 6 | Справочник статусов (ОРЭМ, МТП, ДСД, НТ, АСКУЭ, Резерв) с нормативным временем отсутствия данных |
| МестоРасположения | 10 | Адреса объектов в г. Новый Уренгой |
| Организация | 10 | Организации с признаком потребителя |
| СтруктурноеПодразделение | 10 | Подразделения с привязкой к организациям |
| КонтактноеЛицо | 10 | Контактные лица с должностями, телефонами, email |
| Объект | 10 | Объекты с номерами в БД АльфаЦЕНТР |
| ТочкаУчёта | 10 | ТУ с номерами присоединений, привязкой к объектам и статусам |
| Заявка | 10 | Заявки с датами начала и окончания |
| Примечание | 10 | Текстовые примечания по ТУ |

**Порядок вставки данных** соответствует порядку создания таблиц в DDL-скрипте: сначала наполняются независимые справочники (статусы, места расположения, организации), затем зависимые таблицы. Это гарантирует соблюдение ссылочной целостности при вставке.

**Проверка результатов.** В конце скрипта выполняется анонимный блок (DO), который подсчитывает количество записей в каждой таблице и выводит сводку. Это позволяет оперативно убедиться в корректности наполнения.

**DCL-скрипт настройки безопасности (Приложение Л)**

DCL-скрипт `10_dcl_security.sql` реализует требования раздела 3.4 технического задания (Приложение Д) по разграничению прав доступа и аудиту изменений.

**Ролевая модель.** Созданы три роли доступа:

1. **`amqc_reader`** — роль только для чтения. Назначается всем авторизованным пользователям ПО AlphaMeterQC. Имеет право SELECT на все таблицы схемы `справочная_бд`. Права на новые таблицы, создаваемые в будущем, назначаются автоматически через `ALTER DEFAULT PRIVILEGES`.

2. **`amqc_writer`** — роль для чтения и записи. Назначается только уполномоченным сотрудникам. Имеет права SELECT, INSERT, UPDATE, DELETE на все таблицы, а также USAGE на последовательности (для работы с GENERATED ALWAYS AS IDENTITY).

3. **`amqc_admin`** — роль администратора. Назначается администраторам БД. Имеет полный доступ ко всем объектам схемы, включая создание и изменение объектов.

**Журнал аудита.** Для выполнения требований раздела 3.4.2 технического задания создана таблица `ЖурналАудита` со следующей структурой:

| Столбец | Тип | Описание |
|:--------|:----|:---------|
| ИД_Записи | BIGINT (PK) | Уникальный идентификатор записи аудита |
| ИмяПользователя | VARCHAR(128) | Имя пользователя, выполнившего операцию |
| ВремяИзменения | TIMESTAMP | Время выполнения операции |
| ТипОперации | VARCHAR(10) | INSERT, UPDATE, DELETE |
| ИмяТаблицы | VARCHAR(128) | Имя изменённой таблицы |
| ИД_ЗаписиТаблицы | INTEGER | Идентификатор изменённой записи |
| СтарыеЗначения | JSONB | Значения до изменения (для UPDATE и DELETE) |
| НовыеЗначения | JSONB | Значения после изменения (для INSERT и UPDATE) |

Для автоматической фиксации изменений созданы:
а) **функция `фн_аудит_изменений()`** — триггерная функция на PL/pgSQL, которая в зависимости от типа операции (INSERT, UPDATE, DELETE) формирует запись в журнале аудита с фиксацией старых и новых значений в формате JSONB;
б) **9 триггеров AFTER INSERT OR UPDATE OR DELETE** — по одному на каждую таблицу, вызывающих функцию аудита.

Журнал аудита защищён от удаления и модификации: права DELETE и UPDATE на таблицу `ЖурналАудита` отозваны у ролей `amqc_reader` и `amqc_writer`. Только роль `amqc_admin` имеет полный доступ к журналу.

**Инструкция по развёртыванию и эксплуатации (Приложение М)**

Инструкция `11_deployment_guide.md` содержит полное руководство по развёртыванию и эксплуатации справочной БД, включающее:

а) **требования к окружению** — аппаратные и программные требования (ОС, PostgreSQL 15+, OpenSSL);
б) **порядок развёртывания** — пошаговая инструкция: установка PostgreSQL, создание БД с локалью `ru_RU.UTF-8`, последовательное выполнение DDL, DML и DCL-скриптов, проверка корректности развёртывания;
в) **настройку подключения** — конфигурация pg_hba.conf и postgresql.conf, строка подключения для ПО AlphaMeterQC;
г) **резервное копирование и восстановление** — команды pg_dump/pg_restore, регламент резервного копирования (полное — ежедневно, инкрементальное — непрерывно);
д) **мониторинг и обслуживание** — запросы для проверки активности подключений, анализа размера таблиц, обновления статистики;
е) **устранение неполадок** — типовые проблемы и способы их решения (ошибки подключения, отсутствие таблиц, нарушение внешних ключей);
ж) **контрольный список развёртывания** — 12 пунктов для проверки полноты и корректности развёртывания.
