## 2. Ролевая модель разграничения доступа

### 2.1. Принципы

Разграничение прав доступа к справочной БД AlphaMeterQC основано на следующих принципах:

1. **Принцип минимальных привилегий (POLP)** — каждый пользователь получает только те права, которые необходимы для выполнения его должностных обязанностей.
2. **Разделение обязанностей** — роли чтения и записи разделены; пользователь, имеющий право на изменение данных, не может быть одновременно обычным читающим пользователем.
3. **Административный контроль** — управление правами доступа осуществляется только через роль администратора.

### 2.2. Определение ролей

Созданы три роли доступа (реализованы в DCL-скрипте, Приложение Л):

| Роль | Права | Назначение |
|:-----|:------|:-----------|
| `amqc_reader` | SELECT на все таблицы схемы `справочная_бд` | Все авторизованные пользователи ПО AlphaMeterQC |
| `amqc_writer` | SELECT, INSERT, UPDATE, DELETE на все таблицы; USAGE на последовательности | Уполномоченные сотрудники, ответственные за ведение справочных данных |
| `amqc_admin` | ALL PRIVILEGES на все объекты схемы | Администраторы базы данных |

### 2.3. Матрица доступа

| Объект / Действие | amqc_reader | amqc_writer | amqc_admin |
|:------------------|:-----------:|:-----------:|:----------:|
| SELECT (все таблицы) | ✅ | ✅ | ✅ |
| INSERT (все таблицы) | ❌ | ✅ | ✅ |
| UPDATE (все таблицы) | ❌ | ✅ | ✅ |
| DELETE (все таблицы) | ❌ | ✅ | ✅ |
| USAGE на SEQUENCE | ❌ | ✅ | ✅ |
| CREATE/ALTER/DROP объектов | ❌ | ❌ | ✅ |
| GRANT/REVOKE прав | ❌ | ❌ | ✅ |
| DELETE/UPDATE на ЖурналАудита | ❌ | ❌ | ✅ |

### 2.4. Создание пользователей и назначение ролей

Создание пользователей и назначение ролей выполняется администратором БД. Примеры команд:

```sql
-- Пользователь для чтения (для АРМ инженеров)
CREATE USER amqc_user_reader WITH PASSWORD '***' IN ROLE amqc_reader;

-- Пользователь для записи (для уполномоченных сотрудников)
CREATE USER amqc_user_writer WITH PASSWORD '***' IN ROLE amqc_writer;

-- Пользователь-администратор
CREATE USER amqc_user_admin WITH PASSWORD '***' IN ROLE amqc_admin;
```

Пароли должны соответствовать политике информационной безопасности ООО «Газпром Энерго» (не менее 12 символов, включая буквы верхнего и нижнего регистра, цифры и специальные символы).

## 3. Аудит изменений данных

### 3.1. Журнал аудита

Для фиксации всех изменений данных создана таблица `ЖурналАудита` (Приложение Л, раздел 7.1) со следующей структурой:

| Столбец | Тип | Описание |
|:--------|:----|:---------|
| ИД_Записи | BIGINT (PK) | Уникальный идентификатор записи аудита |
| ИмяПользователя | VARCHAR(128) | Имя пользователя, выполнившего операцию (current_user) |
| ВремяИзменения | TIMESTAMP | Время выполнения операции |
| ТипОперации | VARCHAR(10) | INSERT, UPDATE, DELETE |
| ИмяТаблицы | VARCHAR(128) | Имя изменённой таблицы |
| ИД_ЗаписиТаблицы | INTEGER | Идентификатор изменённой записи |
| СтарыеЗначения | JSONB | Значения до изменения (для UPDATE и DELETE) |
| НовыеЗначения | JSONB | Значения после изменения (для INSERT и UPDATE) |

### 3.2. Триггеры аудита

Для каждой из 9 таблиц создан триггер `AFTER INSERT OR UPDATE OR DELETE`, вызывающий функцию `фн_аудит_изменений()`. Функция в зависимости от типа операции формирует запись в журнале аудита:

а) **INSERT** — фиксируется имя пользователя, время, тип операции, имя таблицы, идентификатор записи и новые значения (JSONB);
б) **UPDATE** — фиксируются старые и новые значения (JSONB) для отслеживания изменений;
в) **DELETE** — фиксируются старые значения (JSONB) для возможности восстановления.

### 3.3. Защита журнала аудита

Журнал аудита защищён от несанкционированного изменения и удаления:

а) Права `DELETE` и `UPDATE` на таблицу `ЖурналАудита` отозваны у ролей `amqc_reader` и `amqc_writer`;
б) Только роль `amqc_admin` имеет полный доступ к журналу;
в) Для ускорения поиска созданы индексы по полям `ВремяИзменения`, `ИмяТаблицы` и `ИмяПользователя`.

### 3.4. Запросы для работы с журналом аудита

**Просмотр последних изменений:**

```sql
SELECT ИмяПользователя, ВремяИзменения, ТипОперации, ИмяТаблицы, ИД_ЗаписиТаблицы
FROM справочная_бд.ЖурналАудита
ORDER BY ВремяИзменения DESC
LIMIT 100;
```

**Просмотр изменений по конкретной таблице:**

```sql
SELECT ИмяПользователя, ВремяИзменения, ТипОперации, ИД_ЗаписиТаблицы,
       СтарыеЗначения, НовыеЗначения
FROM справочная_бд.ЖурналАудита
WHERE ИмяТаблицы = 'ТочкаУчёта'
ORDER BY ВремяИзменения DESC;
```

**Просмотр изменений по конкретному пользователю:**

```sql
SELECT ВремяИзменения, ТипОперации, ИмяТаблицы, ИД_ЗаписиТаблицы
FROM справочная_бд.ЖурналАудита
WHERE ИмяПользователя = 'amqc_user_writer'
ORDER BY ВремяИзменения DESC;
```

## 4. Защита соединений

### 4.1. Требования к защите соединений

В соответствии с требованиями технического задания (Приложение Д, раздел 3.4.3) передача данных между справочной БД и АРМ инженеров должна осуществляться по защищённому протоколу (TLS/SSL) в рамках корпоративной сети.

### 4.2. Настройка TLS/SSL в PostgreSQL

**Генерация сертификатов:**

```bash
# Создание директории для сертификатов
sudo mkdir -p /etc/postgresql/15/main/certs
cd /etc/postgresql/15/main/certs

# Генерация самоподписанного сертификата (для тестовой среды)
sudo openssl req -new -x509 -days 365 -nodes \
    -out server.crt \
    -keyout server.key \
    -subj "/CN=alphameterqc-server"

# Установка прав доступа
sudo chmod 600 server.key
sudo chown postgres:postgres server.crt server.key
```

**Настройка postgresql.conf:**

```
ssl = on
ssl_cert_file = '/etc/postgresql/15/main/certs/server.crt'
ssl_key_file = '/etc/postgresql/15/main/certs/server.key'
ssl_ca_file = ''
```

**Настройка pg_hba.conf для обязательного SSL:**

```
# Подключения с SSL для всех ролей
hostssl    alphameterqc    amqc_reader     192.168.0.0/16    scram-sha-256
hostssl    alphameterqc    amqc_writer     192.168.0.0/16    scram-sha-256
hostssl    alphameterqc    amqc_admin      127.0.0.1/32      scram-sha-256
```

**Перезапуск PostgreSQL:**

```bash
sudo systemctl restart postgresql
```

### 4.3. Строка подключения с SSL (для ПО AlphaMeterQC)

```
postgresql://amqc_reader:password@192.168.1.100:5432/alphameterqc?sslmode=require
```

### 4.4. Сетевая изоляция

а) Сервер БД размещается в выделенном сегменте локальной сети;
б) Доступ к порту 5432 разрешён только с АРМ инженеров (список разрешённых IP-адресов);
в) Доступ к серверу БД по SSH — только для администраторов;
г) Рекомендуется использование файрвола (iptables/nftables) для ограничения доступа.

## 5. Резервное копирование и восстановление

### 5.1. Стратегия резервного копирования

| Тип копирования | Периодичность | Время выполнения | Срок хранения | Инструмент |
|:----------------|:--------------|:-----------------|:--------------|:-----------|
| Полная (custom format) | Ежедневно | 02:00 | 30 дней | pg_dump |
| Непрерывное архивирование WAL | Непрерывно | — | 7 дней | WAL archiving |

### 5.2. Скрипты резервного копирования и восстановления

Разработаны два скрипта (Приложение С):

1. **`16_backup_script.sh`** — скрипт полного резервного копирования в custom-формате с автоматической ротацией (хранение 30 дней). Особенности:
   - проверка доступности PostgreSQL перед началом копирования;
   - ведение журнала операций;
   - автоматическое удаление копий старше 30 дней.

2. **`17_restore_script.sh`** — скрипт восстановления из резервной копии. Особенности:
   - завершение активных подключений к БД перед восстановлением;
   - использование `--clean` и `--if-exists` для пересоздания объектов;
   - ведение журнала операций.

### 5.3. Настройка автоматического резервного копирования (cron)

```bash
# Добавление задачи в crontab
sudo crontab -e

# Ежедневное резервное копирование в 02:00
0 2 * * * /usr/local/bin/backup_alphameterqc.sh >> /var/log/alphameterqc_backup.log 2>&1
```

### 5.4. Процедура восстановления

1. Остановить ПО AlphaMeterQC (завершить подключения к БД).
2. Выполнить скрипт восстановления:
   ```bash
   sudo ./17_restore_script.sh /backup/alphameterqc/alphameterqc_20260504_120000.dump
   ```
3. Проверить целостность восстановленных данных:
   ```sql
   SELECT COUNT(*) AS Таблиц FROM information_schema.tables WHERE table_schema = 'справочная_бд';
   SELECT COUNT(*) AS Записей FROM справочная_бд.ТочкаУчёта;
   ```
4. Запустить ПО AlphaMeterQC.
5. Проверить корректность формирования экранных форм.

## 6. Мониторинг безопасности

### 6.1. Мониторинг попыток несанкционированного доступа

```sql
-- Проверка неудачных попыток подключения (из журнала PostgreSQL)
SELECT count(*) AS Неудачных_подключений
FROM pg_stat_activity
WHERE state = 'failed';
```

### 6.2. Мониторинг активности пользователей

```sql
-- Текущие активные подключения
SELECT usename AS Пользователь, client_addr AS IP_адрес,
       application_name AS Приложение, state AS Состояние,
       query AS Запрос
FROM pg_stat_activity
WHERE datname = 'alphameterqc' AND pid <> pg_backend_pid();
```

### 6.3. Мониторинг размера журнала аудита

```sql
SELECT
    pg_size_pretty(pg_total_relation_size('справочная_бд.ЖурналАудита')) AS РазмерЖурнала,
    COUNT(*) AS КоличествоЗаписей
FROM справочная_бд.ЖурналАудита;
```

## 7. Процедуры реагирования на инциденты

### 7.1. Обнаружение несанкционированного изменения данных

1. Выполнить запрос к журналу аудита для выявления подозрительных операций:
   ```sql
   SELECT * FROM справочная_бд.ЖурналАудита
   WHERE ИмяПользователя NOT IN ('amqc_user_writer', 'amqc_user_admin')
   ORDER BY ВремяИзменения DESC;
   ```

2. При обнаружении несанкционированных изменений — определить объём изменённых данных и временной промежуток.

3. Выполнить восстановление изменённых данных из резервной копии.

### 7.2. Утеря пароля пользователя

1. Администратор БД создаёт новый пароль для пользователя:
   ```sql
   ALTER USER amqc_user_writer WITH PASSWORD 'новый_пароль';
   ```

2. Пользователь обновляет строку подключения в ПО AlphaMeterQC.

### 7.3. Компрометация сертификата TLS

1. Сгенерировать новый сертификат сервера.
2. Обновить файлы сертификатов на сервере БД.
3. Перезапустить PostgreSQL.
4. Обновить сертификаты на всех АРМ инженеров.
