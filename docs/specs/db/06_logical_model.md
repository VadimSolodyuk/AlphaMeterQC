# Логическая модель базы данных
**Модуль справочной БД для ПО AlphaMeterQC**

**Автор:** В.Л. Солодюк
**Проект:** ПО «AlphaMeterQC» (анализ и контроль своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии и хранящихся в БД (Oracle) ПО АльфаЦЕНТР) / Модуль справочной БД

---

## 1. Назначение логической модели

Логическая модель данных является промежуточным звеном между концептуальной ER-диаграммой (Приложение Г) и физической реализацией в СУБД PostgreSQL. Она описывает структуру данных в терминах реляционной модели: таблицы, столбцы, типы данных, ограничения целостности, первичные и внешние ключи, индексы — без привязки к конкретным особенностям реализации СУБД.

## 5. Ограничения целостности

### 5.1. Ограничения NOT NULL
Все столбцы, помеченные как NOT NULL в спецификации таблиц (раздел 2), являются обязательными для заполнения. Нарушение этого требования приводит к ошибке уровня СУБД.

### 5.2. Ограничения UNIQUE
а) `Объект.НомерВБДАльфаЦЕНТР` — уникальный номер объекта в БД АльфаЦЕНТР.
б) `ТочкаУчёта.НомерПрисоединения` — уникальный номер присоединения/фидера.
в) `СтатусТУ.Наименование` — уникальное наименование статуса.

### 5.3. Ограничения DEFAULT
а) `Организация.ПризнакПотребителя` = FALSE (по умолчанию организация не является потребителем).
б) `Примечание.ДатаСоздания` = CURRENT_TIMESTAMP (автоматическая фиксация времени создания).

### 5.4. Ограничения CHECK
а) `СтатусТУ.МаксВремяОтсутствияДанныхЧас > 0` — максимальное время отсутствия данных должно быть положительным числом.

## 8. Нормализация

Логическая модель соответствует требованиям нормальных форм:

а) **Первая нормальная форма (1НФ):** все атрибуты атомарны, каждая таблица имеет первичный ключ, повторяющиеся группы отсутствуют.
б) **Вторая нормальная форма (2НФ):** каждый неключевой атрибут функционально полно зависит от первичного ключа (частичной зависимости нет, так как первичные ключи — простые, одноатрибутные).
в) **Третья нормальная форма (3НФ):** каждый неключевой атрибут нетранзитивно зависит от первичного ключа (все неключевые атрибуты описывают только ту сущность, которой принадлежат).

Детальное описание процесса нормализации приведено в Приложении Ж «Нормализация данных».
