# КОНЦЕПЦИЯ СОЗДАНИЯ ПРОДУКТА / ФИЧИ
**Модуль справочной БД для ПО AlphaMeterQC**

**Автор:** В.Л. Солодюк
**Проект:** ПО «AlphaMeterQC» (анализ и контроль своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии и хранящихся в БД (Oracle) ПО АльфаЦЕНТР) / Модуль справочной БД

---

## 1. Название продукта / фичи
Модуль справочной БД — локальная реляционная база данных, размещаемая на сервере в локальной сети предприятия, для хранения справочной информации о точках учёта (ТУ), объектах электроустановок, организациях, структурных подразделениях, контактных лицах, статусах ТУ, заявках и примечаниях. Используется ПО AlphaMeterQC совместно с данными БД Oracle ПО АльфаЦЕНТР для формирования экранных форм контроля своевременности поступления, полноты и корректности данных со счётчиков электрической энергии.

## 2. Цель создания
Разработать локальную реляционную базу данных для централизованного хранения справочной информации — атрибутов точек учёта, описывающих их с организационно-коммерческой стороны и технико-метрологических параметров. БД используется совместно с данными БД Oracle ПО АльфаЦЕНТР для комплексного анализа своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии, и обеспечения формирования экранных форм в ПО AlphaMeterQC.

**Бизнес-показатели успеха:**
а) сокращение времени формирования экранных форм за счёт централизованного хранения и быстрого доступа к справочным данным;
б) исключение ручного ввода и дублирования справочной информации инженерами при каждом запросе;
в) повышение точности и актуальности отображаемых данных за счёт единого источника справочной информации с возможностью оперативного обновления;
г) обеспечение целостности справочных данных и снижение количества ошибок при формировании отчётности за счёт контроля связей между сущностями на уровне БД.

## 4. Целевая аудитория
а) **Основные пользователи:** Инженеры отдела системного сопровождения — пользователи планируемого настольного ПО AlphaMeterQC.
б) **Демография / сегменты аудитории:** Сотрудники технических специальностей с опытом работы с базами данных, уверенные пользователи ПК. Уровень цифровой грамотности — высокий.
в) **География использования:** Предприятия, на которых внедрена АСКУЭ. Предполагается работа в корпоративной сети предприятия.
г) **Сценарий использования:** Инженер запускает ПО AlphaMeterQC на своём АРМ. Приложение подключается к локальной справочной БД и БД Oracle ПО АльфаЦЕНТР, комбинирует данные для формирования экранных форм. При необходимости инженер через пользовательский интерфейс ПО добавляет, изменяет или удаляет справочные данные (атрибуты ТУ, статусы, заявки, примечания и т.д.), которые сохраняются в локальной БД и используются при следующих запросах.

## 7. Ключевые характеристики продукта / фичи

### 7.1. Функциональные возможности

**Состав хранимых данных:**
а) **Организации:** наименование, признак Потребителя. Организации могут иметь одно и более структурных подразделений.
б) **Структурные подразделения:** наименование, принадлежность к организации. Подразделения могут иметь ноль или более контактных лиц и ноль или более объектов в эксплуатационной ответственности.
в) **Контактные лица:** фамилия, имя, отчество, должность, контактный телефон, электронная почта. Принадлежность к структурному подразделению.
г) **Места расположения:** наименование места расположения объектов.
д) **Объекты/электроустановки:** наименование, уникальный номер в БД АльфаЦЕНТР, место расположения, структурное подразделение-ответственный.
е) **Точки учёта (ТУ):** наименование, уникальный номер присоединения/фидера в БД АльфаЦЕНТР, принадлежность к объекту, статус.
ж) **Статусы ТУ:** наименование (ОРЭМ, МТП, ДСД и др.), максимально допустимое время отсутствия поступления данных в БД АльфаЦЕНТР (в часах).
з) **Заявки по ТУ:** номер заявки, дата начала, дата окончания. Привязка к конкретной ТУ.
и) **Примечания по ТУ:** текст примечания, дата создания. Привязка к конкретной ТУ.

**Операции управления данными:**
а) добавление новых записей во все справочники;
б) редактирование существующих записей;
в) удаление записей с обеспечением контроля ссылочной целостности;
г) каскадное удаление связанных записей (при удалении организации удаляются её подразделения; при удалении объекта — его ТУ; при удалении ТУ — её заявки и примечания);
д) запрет удаления записей, на которые имеются ссылки (статус ТУ не может быть удалён, если есть ТУ с этим статусом; место расположения не может быть удалено, если в нём есть объекты).

**Интеграция с ПО AlphaMeterQC:**
а) предоставление программного интерфейса (API) для выполнения операций чтения и записи справочных данных;
б) возможность выполнения SQL-запросов для комбинирования данных справочной БД с данными БД АльфаЦЕНТР;
в) использование открытого стандартизированного диалекта SQL.

**Аудит и безопасность:**
а) разграничение прав доступа: чтение — всем авторизованным пользователям, запись — только уполномоченным сотрудникам;
б) ведение журнала изменений данных (аудит) с фиксацией: кто, когда и какие изменения внёс;
в) передача данных по защищённому протоколу в рамках корпоративной сети.

### 7.2. Нефункциональные требования
а) **СУБД:** Реляционная СУБД с открытым исходным кодом (PostgreSQL).
б) **Размещение:** Сервер в локальной сети предприятия.
в) **Режим работы:** Многопользовательский, с возможностью одновременного подключения нескольких АРМ инженеров.
г) **Доступность:** Не менее 99,5% в круглосуточном режиме 24/7.
д) **Производительность:** Время выполнения типового запроса на чтение (список ТУ по объекту, список заявок по ТУ) — не более 1 секунды при объёме данных до 10 000 записей.
е) **Резервное копирование:** Поддержка процедур резервного копирования и восстановления данных без остановки работы приложения.
ж) **Развёртывание:** Документированный процесс, включающий скрипты создания схемы данных, начального наполнения справочников и настройки прав доступа.
з) **Целостность данных:** Обеспечение ссылочной целостности на уровне БД (внешние ключи, каскадные операции, ограничения).

## 8. Уникальные преимущества (USP)
а) **Целенаправленность:** Модуль решает ровно одну задачу — централизованное хранение справочной информации о точках учёта — без избыточного функционала.
б) **Целостность по умолчанию:** Ссылочная целостность данных обеспечивается на уровне БД (внешние ключи, каскадные операции), что исключает появление «осиротевших» записей.
в) **Совместимость с БД АльфаЦЕНТР:** Структура данных спроектирована для бесшовного комбинирования с данными БД Oracle ПО АльфаЦЕНТР через стандартные SQL-запросы.
г) **Открытый стек:** Использование PostgreSQL — открытой, широко распространённой СУБД с богатыми возможностями администрирования и резервного копирования.
д) **Прозрачность изменений:** Встроенный аудит изменений обеспечивает полную отслеживаемость всех операций со справочными данными.

## 10. Ожидаемые результаты и метрики успеха
**Бизнес-метрики:**
а) сокращение времени формирования экранной формы с 15–30 минут до 1 минуты;
б) исключение ошибок, связанных с разрозненным хранением справочных данных;
в) снижение времени на поиск контактной информации и истории заявок по ТУ до нескольких секунд.

**Ключевые критерии приёмки:**
1. БД содержит все требуемые сущности: организации, подразделения, контактные лица, места расположения, объекты, ТУ, статусы ТУ, заявки, примечания.
2. Между сущностями установлены связи с правилами каскадного удаления и/или запрета удаления в соответствии с требованиями.
3. Для каждой сущности определён первичный ключ, для связанных сущностей — внешние ключи.
4. Время выполнения типового запроса на чтение не превышает 1 секунды при объёме данных до 10 000 записей.
5. Разработаны и документированы SQL-скрипты создания схемы данных, начального наполнения и настройки прав доступа.
6. Настроено резервное копирование с возможностью восстановления данных.
7. Реализовано разграничение прав доступа (чтение/запись).
8. Ведётся журнал аудита изменений справочных данных.
9. БД развёрнута на PostgreSQL и доступна для подключения с нескольких АРМ.
10. Подготовлена документация по развёртыванию и эксплуатации БД.
