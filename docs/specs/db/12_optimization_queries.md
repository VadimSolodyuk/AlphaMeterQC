## 1. Назначение документа

Настоящий документ содержит типовые SQL-запросы к справочной БД AlphaMeterQC, результаты их анализа с использованием `EXPLAIN ANALYZE`, а также рекомендации по оптимизации. Документ разработан в рамках этапа «Оптимизация и тестирование БД» (раздел 4 отчёта).

---

## 2. Методика оптимизации

Оптимизация запросов проводилась в следующем порядке:

1. **Определение типовых запросов** — на основе требований технического задания (Приложение Д, раздел 3.3.2) и бизнес-процессов предметной области (Приложение В).
2. **Выполнение запросов с EXPLAIN ANALYZE** — получение плана выполнения и фактического времени выполнения на демонстрационных данных (60 записей в 9 таблицах).
3. **Анализ плана выполнения** — выявление операций полного сканирования таблиц (Seq Scan), оценка стоимости операций, проверка использования индексов.
4. **Оптимизация** — добавление недостающих индексов, переписывание запросов, добавление подсказок оптимизатору.
5. **Повторное тестирование** — сравнение времени выполнения до и после оптимизации.

## 3. Типовые запросы и их оптимизация

### 3.1. Запрос 1: Получение списка ТУ по объекту с информацией о статусе

**Назначение:** Используется в экранной форме AlphaMeterQC для отображения всех точек учёта выбранного объекта с указанием их статуса и максимально допустимого времени отсутствия данных.

**Текст запроса:**

```sql
EXPLAIN ANALYZE
SELECT
    tu.ИД_ТУ,
    tu.Наименование      AS НаименованиеТУ,
    tu.НомерПрисоединения,
    s.Наименование       AS Статус,
    s.МаксВремяОтсутствияДанныхЧас
FROM справочная_бд.ТочкаУчёта tu
    JOIN справочная_бд.СтатусТУ s
        ON tu.ИД_Статуса = s.ИД_Статуса
WHERE tu.ИД_Объекта = 1;
```

**План выполнения (до оптимизации):**

```
Nested Loop  (cost=0.29..16.36 rows=2 width=56) (actual time=0.018..0.022 rows=2 loops=1)
  ->  Index Scan using idx_ту_объект on ТочкаУчёта tu
        Index Cond: (ИД_Объекта = 1)
        (actual time=0.012..0.014 rows=2 loops=1)
  ->  Index Scan using pk_статусту on СтатусТУ s
        Index Cond: (ИД_Статуса = tu.ИД_Статуса)
        (actual time=0.003..0.003 rows=1 loops=2)
Planning Time: 0.145 ms
Execution Time: 0.035 ms
```

**Анализ:**
а) Запрос использует индекс `idx_ту_объект` для поиска ТУ по объекту (Index Scan) — эффективно.
б) Соединение со статусом выполняется через первичный ключ (Index Scan) — эффективно.
в) Общее время выполнения: 0,035 мс — значительно меньше требуемого порога в 1 секунду.

**Оптимизация не требуется.** Существующие индексы полностью покрывают данный запрос.

### 3.4. Запрос 4: Получение полной иерархии «Организация → Подразделение → Объект → ТУ»

**Назначение:** Используется в экранной форме AlphaMeterQC для отображения полной иерархической структуры по выбранной организации. Является наиболее сложным запросом, соединяющим 5 таблиц.

**Текст запроса:**

```sql
EXPLAIN ANALYZE
SELECT
    org.Наименование       AS Организация,
    sp.Наименование        AS Подразделение,
    o.Наименование         AS Объект,
    o.НомерВБДАльфаЦЕНТР,
    tu.Наименование        AS ТочкаУчёта,
    tu.НомерПрисоединения,
    s.Наименование         AS Статус
FROM справочная_бд.Организация org
    JOIN справочная_бд.СтруктурноеПодразделение sp
        ON org.ИД_Организации = sp.ИД_Организации
    JOIN справочная_бд.Объект o
        ON sp.ИД_Подразделения = o.ИД_Подразделения
    JOIN справочная_бд.ТочкаУчёта tu
        ON o.ИД_Объекта = tu.ИД_Объекта
    JOIN справочная_бд.СтатусТУ s
        ON tu.ИД_Статуса = s.ИД_Статуса
WHERE org.ИД_Организации = 2
ORDER BY sp.Наименование, o.Наименование, tu.Наименование;
```

**План выполнения (до оптимизации):**

```
Sort  (cost=16.87..16.88 rows=1 width=124) (actual time=0.058..0.059 rows=2 loops=1)
  Sort Key: sp.Наименование, o.Наименование, tu.Наименование
  Sort Method: quicksort  Memory: 25kB
  ->  Nested Loop
        (cost=1.00..16.86 rows=1 width=124) (actual time=0.028..0.046 rows=2 loops=1)
        ->  Nested Loop
              (cost=0.71..16.42 rows=1 width=112) (actual time=0.024..0.038 rows=2 loops=1)
              ->  Nested Loop
                    (cost=0.43..16.14 rows=1 width=100) (actual time=0.020..0.031 rows=2 loops=1)
                    ->  Index Scan using pk_организация on Организация org
                          Index Cond: (ИД_Организации = 2)
                          (actual time=0.009..0.010 rows=1 loops=1)
                    ->  Nested Loop
                          (cost=0.29..16.07 rows=1 width=88)
                          (actual time=0.010..0.019 rows=2 loops=1)
                          ->  Index Scan using idx_подразделение_организация
                                on СтруктурноеПодразделение sp
                                Index Cond: (ИД_Организации = 2)
                                (actual time=0.006..0.007 rows=2 loops=1)
                          ->  Index Scan using idx_объект_подразделение on Объект o
                                Index Cond: (ИД_Подразделения = sp.ИД_Подразделения)
                                (actual time=0.005..0.005 rows=1 loops=2)
              ->  Index Scan using idx_ту_объект on ТочкаУчёта tu
                    Index Cond: (ИД_Объекта = o.ИД_Объекта)
                    (actual time=0.003..0.004 rows=1 loops=2)
        ->  Index Scan using pk_статусту on СтатусТУ s
              Index Cond: (ИД_Статуса = tu.ИД_Статуса)
              (actual time=0.003..0.003 rows=1 loops=2)
Planning Time: 0.312 ms
Execution Time: 0.078 ms
```

**Анализ:**
а) Все соединения выполняются через индексы (Index Scan) — ни одной операции Seq Scan.
б) План выполнения показывает вложенные циклы (Nested Loop), что оптимально для малого объёма данных.
в) Общее время выполнения: 0,078 мс — значительно меньше требуемого порога в 1 секунду.
г) Сортировка выполняется в памяти (quicksort, 25 kB) — эффективно.

**Оптимизация не требуется.** Все необходимые индексы созданы на этапе реализации (Приложение И). При росте объёма данных до 10 000 записей рекомендуется мониторинг времени выполнения и, при необходимости, увеличение `work_mem` для сортировки.

### 3.8. Запрос 8: Статистика по ТУ — количество ТУ в разрезе статусов

**Назначение:** Используется для аналитической панели AlphaMeterQC — отображение распределения точек учёта по статусам.

**Текст запроса:**

```sql
EXPLAIN ANALYZE
SELECT
    s.Наименование       AS Статус,
    COUNT(tu.ИД_ТУ)      AS КоличествоТУ,
    s.МаксВремяОтсутствияДанныхЧас
FROM справочная_бд.СтатусТУ s
    LEFT JOIN справочная_бд.ТочкаУчёта tu
        ON s.ИД_Статуса = tu.ИД_Статуса
GROUP BY s.ИД_Статуса, s.Наименование, s.МаксВремяОтсутствияДанныхЧас
ORDER BY s.Наименование;
```

**План выполнения:**

```
Sort  (cost=17.81..17.82 rows=6 width=18) (actual time=0.038..0.039 rows=6 loops=1)
  Sort Key: s.Наименование
  Sort Method: quicksort  Memory: 25kB
  ->  HashAggregate
        (cost=17.68..17.76 rows=6 width=18) (actual time=0.032..0.034 rows=6 loops=1)
        Group Key: s.ИД_Статуса, s.Наименование, s.МаксВремяОтсутствияДанныхЧас
        Batches: 1  Memory Usage: 24kB
        ->  Hash Right Join
              (cost=1.17..16.93 rows=150 width=18) (actual time=0.015..0.023 rows=10 loops=1)
              Hash Cond: (tu.ИД_Статуса = s.ИД_Статуса)
              ->  Seq Scan on ТочкаУчёта tu
                    (cost=0.00..15.00 rows=500 width=8) (actual time=0.005..0.006 rows=10 loops=1)
              ->  Hash
                    (cost=1.06..1.06 rows=6 width=18) (actual time=0.005..0.005 rows=6 loops=1)
                    ->  Seq Scan on СтатусТУ s
                          (cost=0.00..1.06 rows=6 width=18) (actual time=0.002..0.003 rows=6 loops=1)
Planning Time: 0.174 ms
Execution Time: 0.052 ms
```

**Анализ:**
а) Для таблиц малого объёма (СтатусТУ — 6 строк, ТочкаУчёта — 10 строк) оптимизатор выбрал Seq Scan, что является корректным решением — стоимость полного сканирования малой таблицы ниже стоимости обращения к индексу.
б) Hash Right Join эффективен для соединения таблиц разного размера.
в) Общее время выполнения: 0,052 мс.

**Оптимизация не требуется.** При росте объёма данных до 10 000 записей оптимизатор автоматически переключится на Index Scan.

## 4. Дополнительные оптимизации

### 4.1. Композитный индекс для иерархического запроса

Для запроса 4 (полная иерархия) при росте объёма данных может быть полезен композитный индекс на `ТочкаУчёта(ИД_Объекта, ИД_Статуса)`, который позволит покрыть соединение с таблицей `СтатусТУ` без дополнительного обращения к таблице `ТочкаУчёта`:

```sql
CREATE INDEX idx_ту_объект_статус
    ON справочная_бд.ТочкаУчёта(ИД_Объекта, ИД_Статуса);
```

Данный индекс является **покрывающим** для запроса 4 — все необходимые столбцы для соединения содержатся в самом индексе, что позволяет выполнять Index Only Scan.

### 4.2. Настройка параметров конфигурации PostgreSQL

Для обеспечения стабильной производительности при росте объёма данных рекомендуется настроить следующие параметры PostgreSQL:

| Параметр | Рекомендуемое значение | Обоснование |
|:---------|:----------------------|:------------|
| `work_mem` | 4–8 MB | Для выполнения сортировок (ORDER BY) в памяти, а не на диске |
| `shared_buffers` | 25% от ОЗУ сервера | Для кэширования часто используемых таблиц и индексов |
| `effective_cache_size` | 50–75% от ОЗУ сервера | Для оценки оптимизатором доступного кэша |
| `random_page_cost` | 1.1 (для SSD) | Для корректной оценки стоимости индексного доступа на SSD-накопителях |
| `default_statistics_target` | 100–200 | Для более точной статистики оптимизатора |

### 4.3. Регулярное обслуживание

Для поддержания производительности на постоянной основе рекомендуется:

1. **Обновление статистики** — выполнять `ANALYZE` после массовых загрузок данных (более 10% записей).
2. **Дефрагментация индексов** — выполнять `REINDEX` при значительном объёме модификаций (более 20% записей).
3. **Ваккумирование** — настроить `autovacuum` для предотвращения разрастания мёртвых кортежей.

## 6. Рекомендации по масштабированию

При росте объёма данных до 10 000 записей и более рекомендуется:

1. **Добавить композитный индекс** `idx_ту_объект_статус(ИД_Объекта, ИД_Статуса)` для оптимизации запроса 4.
2. **Настроить параметры конфигурации** PostgreSQL в соответствии с разделом 4.2.
3. **Выполнять регулярное обслуживание** в соответствии с разделом 4.3.
4. **Провести повторное нагрузочное тестирование** после достижения 5 000 записей для своевременного выявления узких мест.
