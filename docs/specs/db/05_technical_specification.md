## 2. Назначение и цели создания базы данных

### 2.1. Назначение базы данных
Справочная БД AlphaMeterQC предназначена для централизованного хранения справочной информации о точках учёта электрической энергии, объектах электроустановок, организациях, структурных подразделениях, контактных лицах, статусах точек учёта, заявках и примечаниях. База данных используется совместно с данными БД Oracle ПО «АльфаЦЕНТР» для формирования экранных форм в ПО AlphaMeterQC, обеспечивающих контроль своевременности поступления, полноты и корректности данных со счётчиков электрической энергии.

### 2.2. Цели создания базы данных
Целями создания справочной БД являются:
1. **Централизация хранения** — устранение разрозненности справочных данных, исключение дублирования и потери информации.
2. **Автоматизация формирования экранных форм** — сокращение времени формирования экранных форм с 15–30 минут до 1 минуты за счёт автоматического комбинирования справочных данных с данными БД АльфаЦЕНТР.
3. **Обеспечение целостности данных** — предотвращение ошибок в отчётности за счёт контроля ссылочной целостности на уровне БД.
4. **Оперативность обновления** — обеспечение возможности своевременного добавления, изменения и удаления справочных данных через интерфейс ПО AlphaMeterQC.
5. **Контроль доступа** — реализация разграничения прав доступа к справочным данным в соответствии с политикой информационной безопасности.
6. **Сохранность данных** — обеспечение возможности резервного копирования и восстановления данных при сбоях.

## 3. Требования к базе данных

### 3.1. Требования к составу и структуре данных

#### 3.1.1. Перечень хранимых сущностей
База данных должна обеспечивать хранение следующего состава справочных данных:

**3.1.1.1. Организации**
а) Наименование организации (обязательное поле).
б) Признак Потребителя (флаг, указывающий, является ли организация потребителем электрической энергии).
в) Организации могут иметь одно и более структурных подразделений.

**3.1.1.2. Структурные подразделения**
а) Наименование подразделения (обязательное поле).
б) Принадлежность к организации (обязательная ссылка).
в) Подразделения могут иметь ноль или более контактных лиц.
г) Подразделения могут иметь ноль или более объектов в эксплуатационной ответственности.

**3.1.1.3. Контактные лица**
а) Фамилия (обязательное поле).
б) Имя.
в) Отчество.
г) Должность.
д) Контактный телефон.
е) Электронная почта.
ж) Принадлежность к структурному подразделению (обязательная ссылка).

**3.1.1.4. Места расположения**
а) Наименование места расположения (обязательное поле).
б) В одном месте расположения может находиться более одного объекта.

**3.1.1.5. Объекты/электроустановки**
а) Наименование объекта (обязательное поле).
б) Уникальный номер объекта в БД АльфаЦЕНТР (обязательное поле).
в) Место расположения (обязательная ссылка).
г) Структурное подразделение, в эксплуатационной ответственности которого находится объект (обязательная ссылка).
д) Каждый объект относится ровно к одному месту расположения и ровно к одному структурному подразделению.

**3.1.1.6. Точки учёта (ТУ)**
а) Наименование ТУ (обязательное поле).
б) Уникальный номер присоединения/фидера в БД АльфаЦЕНТР (обязательное поле).
в) Принадлежность к объекту (обязательная ссылка).
г) Статус ТУ (обязательная ссылка).
д) Каждая ТУ относится ровно к одному объекту.

**3.1.1.7. Статусы ТУ**
а) Наименование статуса (ОРЭМ, МТП, ДСД и др.) (обязательное поле).
б) Максимально допустимое время отсутствия поступления данных в БД АльфаЦЕНТР (в часах) (обязательное поле).

**3.1.1.8. Заявки по ТУ**
а) Номер заявки (обязательное поле).
б) Дата начала (обязательное поле).
в) Дата окончания.
г) Привязка к конкретной ТУ (обязательная ссылка).
д) Одна ТУ может иметь несколько заявок.

**3.1.1.9. Примечания по ТУ**
а) Текст примечания (обязательное поле).
б) Дата создания.
в) Привязка к конкретной ТУ (обязательная ссылка).
г) Одна ТУ может иметь несколько примечаний.

#### 3.1.2. Требования к первичным ключам
а) Каждая таблица должна иметь первичный ключ — суррогатный целочисленный идентификатор (тип INTEGER).
б) Наименование первичного ключа: `ИД_<Сущность>` (например, `ИД_Организации`, `ИД_Объекта`).
в) Значения первичных ключей должны автоматически генерироваться СУБД (SEQUENCE + DEFAULT или GENERATED ALWAYS AS IDENTITY).

#### 3.1.3. Требования к внешним ключам и ссылочной целостности
Для каждой связи между сущностями должен быть определён внешний ключ с явно заданным правилом каскадного удаления:

| Связь | Внешний ключ | Правило удаления |
|-------|-------------|-------------------|
| Организация → СтруктурноеПодразделение | `ИД_Организации` в таблице `СтруктурноеПодразделение` | CASCADE |
| СтруктурноеПодразделение → КонтактноеЛицо | `ИД_Подразделения` в таблице `КонтактноеЛицо` | CASCADE |
| СтруктурноеПодразделение → Объект | `ИД_Подразделения` в таблице `Объект` | RESTRICT |
| МестоРасположения → Объект | `ИД_МестаРасположения` в таблице `Объект` | RESTRICT |
| Объект → ТочкаУчёта | `ИД_Объекта` в таблице `ТочкаУчёта` | CASCADE |
| СтатусТУ → ТочкаУчёта | `ИД_Статуса` в таблице `ТочкаУчёта` | RESTRICT |
| ТочкаУчёта → Заявка | `ИД_ТУ` в таблице `Заявка` | CASCADE |
| ТочкаУчёта → Примечание | `ИД_ТУ` в таблице `Примечание` | CASCADE |

#### 3.1.4. Требования к ограничениям целостности
а) Все обязательные поля (NOT NULL) должны быть явно объявлены.
б) Для полей, имеющих уникальность по бизнес-логике (например, `НомерПрисоединения` в таблице `ТочкаУчёта`), должны быть созданы ограничения UNIQUE.
в) Для поля `ПризнакПотребителя` в таблице `Организация` должно быть задано значение по умолчанию FALSE.
г) Для поля `ДатаСоздания` в таблице `Примечание` должно быть задано значение по умолчанию CURRENT_TIMESTAMP.

### 3.2. Требования к производительности

#### 3.2.1. Время выполнения запросов
а) Время выполнения типового запроса на чтение (список ТУ по объекту, список заявок по ТУ) не должно превышать 1 секунды при объёме данных до 10 000 записей.
б) Время выполнения операции вставки одной записи не должно превышать 0,5 секунды.
в) Время выполнения операции обновления одной записи не должно превышать 0,5 секунды.

#### 3.2.2. Требования к индексам
Для обеспечения производительности запросов должны быть созданы следующие индексы:
а) Индекс по полю `ИД_Объекта` в таблице `ТочкаУчёта` — для ускорения поиска ТУ по объекту.
б) Индекс по полю `ИД_ТУ` в таблице `Заявка` — для ускорения поиска заявок по ТУ.
в) Индекс по полю `ИД_ТУ` в таблице `Примечание` — для ускорения поиска примечаний по ТУ.
г) Индекс по полю `ИД_Подразделения` в таблице `Объект` — для ускорения поиска объектов по подразделению.
д) Индекс по полю `НомерПрисоединения` в таблице `ТочкаУчёта` — для ускорения поиска ТУ по номеру присоединения.

### 3.3. Требования к программному интерфейсу (API)

#### 3.3.1. Режимы доступа к данным
а) Доступ к данным должен осуществляться через стандартные SQL-запросы (диалект SQL, поддерживаемый выбранной СУБД).
б) ПО AlphaMeterQC должно иметь возможность выполнять запросы SELECT, INSERT, UPDATE, DELETE ко всем таблицам справочной БД.

#### 3.3.2. Типовые запросы
Должны быть разработаны и документированы следующие типовые запросы:
а) Получение списка ТУ по объекту с информацией о статусе.
б) Получение списка заявок по ТУ.
в) Получение списка примечаний по ТУ.
г) Получение полной иерархии: организация → подразделение → объект → ТУ.
д) Получение контактных лиц подразделения, в эксплуатационной ответственности которого находится объект.

### 3.4. Требования к безопасности

#### 3.4.1. Разграничение прав доступа
а) Должны быть созданы как минимум две роли: `reader` (чтение всех таблиц) и `writer` (чтение, вставка, обновление, удаление во всех таблицах).
б) Роль `reader` должна быть назначена всем авторизованным пользователям ПО AlphaMeterQC.
в) Роль `writer` должна быть назначена только уполномоченным сотрудникам.

#### 3.4.2. Аудит изменений
а) Должен вестись журнал аудита изменений данных с фиксацией: идентификатора пользователя, времени изменения, типа операции (INSERT, UPDATE, DELETE), изменённой таблицы и идентификатора изменённой записи.
б) Журнал аудита должен быть защищён от удаления и модификации.

#### 3.4.3. Защита соединений
а) Передача данных между справочной БД и АРМ инженеров должна осуществляться по защищённому протоколу (TLS/SSL) в рамках корпоративной сети.

### 3.5. Требования к надёжности

#### 3.5.1. Режим работы
а) База данных должна обеспечивать стабильную работу в круглосуточном режиме 24/7.
б) Доступность БД должна составлять не менее 99,5%.

#### 3.5.2. Резервное копирование и восстановление
а) Должна быть обеспечена возможность резервного копирования данных без остановки работы приложения.
б) Должна быть обеспечена возможность восстановления данных из резервной копии.
в) Должен быть разработан регламент резервного копирования с указанием периодичности, типа копирования (полное/инкрементальное) и срока хранения копий.

### 3.6. Требования к развёртыванию

#### 3.6.1. Выбор СУБД
а) В качестве СУБД должна использоваться реляционная система управления базами данных с открытым исходным кодом — PostgreSQL.

#### 3.6.2. Размещение
а) База данных должна быть развёрнута на сервере в локальной сети предприятия.

#### 3.6.3. Многопользовательский режим
а) База данных должна поддерживать многопользовательский режим работы с возможностью одновременного подключения нескольких АРМ инженеров.

#### 3.6.4. Документация развёртывания
а) Процесс развёртывания БД должен быть документирован и включать:
   1) скрипты создания схемы данных (DDL);
   2) скрипты начального наполнения справочников (DML);
   3) скрипты настройки прав доступа (DCL);
   4) инструкцию по развёртыванию.

## 6. Порядок контроля и приёмки

### 6.1. Критерии приёмки
База данных считается готовой к эксплуатации при выполнении следующих критериев:

1. **Полнота структуры данных:** БД содержит все 9 требуемых сущностей (Организация, СтруктурноеПодразделение, КонтактноеЛицо, МестоРасположения, Объект, ТочкаУчёта, СтатусТУ, Заявка, Примечание) с полным набором атрибутов.

2. **Целостность связей:** Между сущностями установлены связи с правилами каскадного удаления (CASCADE) и/или запрета удаления (RESTRICT) в соответствии с требованиями раздела 3.1.3.

3. **Первичные и внешние ключи:** Для каждой сущности определён первичный ключ, для связанных сущностей — внешние ключи.

4. **Производительность:** Время выполнения типового запроса на чтение не превышает 1 секунды при объёме данных до 10 000 записей.

5. **Документирование:** Разработаны и документированы SQL-скрипты создания схемы данных, начального наполнения и настройки прав доступа.

6. **Резервное копирование:** Настроено резервное копирование с возможностью восстановления данных.

7. **Безопасность:** Реализовано разграничение прав доступа (роли reader/writer), ведётся журнал аудита изменений.

8. **Развёртывание:** БД развёрнута на PostgreSQL и доступна для подключения с нескольких АРМ.

### 6.2. Порядок предъявления результатов
Результаты работ предъявляются руководителю практики в виде:
а) настоящего технического задания;
б) разработанных SQL-скриптов;
в) развёрнутой и настроенной базы данных;
г) отчёта о прохождении практики.

## 7. Источники разработки

При разработке технического задания использовались следующие нормативные и справочные материалы:
а) ГОСТ 34.602-2020 «Информационная технология. Комплекс стандартов на автоматизированные системы. Техническое задание на создание автоматизированной системы»;
б) ГОСТ Р 59792-2021 «Информационные технологии. Комплекс стандартов на автоматизированные системы. Виды испытаний автоматизированных систем»;
в) Документация по программному комплексу «АльфаЦЕНТР» (внутренняя документация ООО «Газпром Энерго»);
г) Политика информационной безопасности ООО «Газпром Энерго» (внутренний документ).
