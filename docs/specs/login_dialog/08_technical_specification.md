## 1. ОСНОВАНИЕ ДЛЯ РАЗРАБОТКИ
Основанием для разработки является задание на прохождение производственной практики в составе профессионального модуля ПМ.02 «Осуществление интеграции программных модулей».  
План-график работ приведён в Приложении А.  
Перечень нормативно-технических документов приведён в Приложении В.

## 2. НАЗНАЧЕНИЕ РАЗРАБОТКИ
Модуль `login_dialog` предназначен для ввода пользователем идентификационных данных для подключения к БД Oracle, валидации формата вводимых данных, маскировки пароля и передачи данных вызывающей системе (ПО «AlphaMeterQC»). Модуль не выполняет самостоятельного подключения к БД и не проверяет достоверность учётных данных на сервере.

## 3. АРХИТЕКТУРА МОДУЛЯ
Модуль реализует архитектурный паттерн **MVC (Model-View-Controller)**:

| Компонент | Реализация | Ответственность |
|-----------|------------|-----------------|
| **Model** | `ConnectionConfig`, `Validator` | Загрузка/сохранение JSON, правила валидации |
| **View** | `ConnectionDialog` | Графический интерфейс (CustomTkinter) |
| **Controller** | `LoginDialog` | API вызова, координация Model и View |

**Режимы интеграции:** Для исключения конфликтов графических циклов (Event Loop) модуль поддерживает два режима работы:
1. **Режим библиотеки (по умолчанию):** Модуль импортируется как Python-пакет, функция `show_dialog()` вызывается напрямую. Рекомендуется, если вызывающая система использует Tkinter/CustomTkinter.
2. **Режим отдельного процесса (альтернативный):** Модуль упаковывается в самостоятельный `.exe` файл. Вызывающая система запускает его через `subprocess`. Результат передаётся через `stdout` в формате JSON. Гарантирует совместимость с любой вызывающей системой (PyQt, PySide, C++, C# и т.д.).

**Инициализация:** При импорте модуля GUI не инициализируется. Инициализация происходит при первом вызове `show_dialog()`. `Validator` создаётся при инициализации `ConnectionDialog`.  
**Конкурентный доступ к файлу:** При одновременной работе нескольких экземпляров модуля последний экземпляр перезаписывает файл. Блокировки не требуются.  
**Потокобезопасность:** Все вызовы GUI должны выполняться в главном потоке процесса. Модуль не создаёт дополнительных фоновых потоков.  
**Обработка исключений:** Модуль не генерирует необработанных исключений наружу. Все ошибки файловой системы обрабатываются внутри модуля молча.

## 4. ТРЕБОВАНИЯ К ПРОГРАММНОМУ МОДУЛЮ

### 4.1. Требования к функциональным характеристикам

#### 4.1.1. Отображение графического интерфейса (F-1)
Модуль должен отображать графическое окно со следующими элементами (реализовано на базе CustomTkinter):

| Элемент | Тип | Подпись на русском языке | Обоснование |
|---------|-----|-------------------------|-------------|
| IP-адрес / Хост | Поле ввода | «IP-адрес/хост» | SR-1, US-1 |
| Порт | Поле ввода | «Порт» | SR-1, US-1 |
| Имя пользователя | Поле ввода | «Имя пользователя» | SR-1, US-1 |
| Пароль | Поле ввода (маскированное) | «Пароль» | SR-1, US-1 |
| Идентификатор службы | Поле ввода | «Идентификатор службы» | SR-1, US-1 |
| Ок | Кнопка | «Ок» | SR-1, US-1 |
| Отмена | Кнопка | «Отмена» | SR-1, US-1 |

**Порядок навигации клавишей Tab:**  
IP-адрес/хост → Порт → Имя пользователя → Пароль → Идентификатор службы → «Ок» → «Отмена».

**Обработка нажатия Enter:**  
Если форма валидна (кнопка «Ок» активна), нажатие Enter в любом поле ввода равносильно нажатию кнопки «Ок».  
Если форма невалидна (кнопка «Ок» неактивна), нажатие Enter игнорируется, окно не закрывается.

**Закрытие окна:** Закрытие окна через системную кнопку «X» равносильно нажатию кнопки «Отмена» (возврат `None`).

#### 4.1.2. Валидация вводимых данных (F-2)
Модуль должен выполнять проверку корректности полей в реальном времени при изменении содержимого.

**Правила валидации:**

| Поле | Правило | Обоснование |
|------|---------|-------------|
| IP-адрес / Хост | Валидный IPv4-адрес (0-255) **или** DNS-имя хоста. Для DNS: строка после удаления пробелов по краям (`strip`) должна содержать хотя бы один буквенно-цифровой символ, не начинаться и не заканчиваться на '-' или '.', допустимые символы: латиница, цифры, '_', '-', '.', не более 255 символов. | SR-2, US-2 |
| Порт | Число от 1 до 65535 | SR-2, US-2 |
| Имя пользователя | Не пустое, ≤ 30 символов (латиница, цифры, '_', '-') | SR-2, US-2 |
| Пароль | Не пустой (≥ 1 символа, любые символы). Длина не ограничена | SR-2, US-2 |
| Идентификатор службы | ≤ 30 символов (латиница, цифры, '_', '.', '-'). Поле опционально | SR-2, US-2 |

Метод `validateAll()` возвращает `bool` — `True`, если все поля валидны.  
**Обработка недопустимых символов:** Недопустимые символы не блокируются на уровне ввода. Поле подсвечивается, кнопка «Ок» остаётся неактивной до исправления.

**Граничные значения для тестирования:**

| Поле | Мин. | Макс. | Негативные значения |
|------|------|-------|---------------------|
| IP-адрес/Хост (IPv4) | `0.0.0.0` | `255.255.255.255` | `256.1.1.1`, `1.2.3`, пустая строка, кириллица |
| IP-адрес/Хост (DNS) | `a` (1 символ) | 255 символов | `---`, `...`, `oracle..db.local` (двойная точка), строка из 256 символов, кириллица |
| Порт | 1 | 65535 | 0, 65536, «abc», пустая строка |
| Имя пользователя | 1 символ | 30 символов | пустая строка, 31 символ, спецсимволы |
| Пароль | 1 символ | не ограничен | пустая строка |
| Идентификатор службы | 0 символов | 30 символов | 31 символ, недопустимые символы |

При невыполнении правила поле должно визуально подсвечиваться (F-3).

#### 4.1.3. Управление кнопкой «Ок» (F-4)
Кнопка «Ок» должна быть неактивна, пока любое обязательное поле пустое или не соответствует правилам проверки. Кнопка «Отмена» должна быть активна всегда.

#### 4.1.4. Загрузка сохранённых параметров (F-8, F-13)
При запуске модуль должен читать JSON-файл из локальной защищённой области пользователя по пути:

| Операционная система | Путь к файлу |
|---------------------|--------------|
| Windows | `%LOCALAPPDATA%\alphameterqc\connection.json` |
| Linux | `~/.config/alphameterqc/connection.json` |

**Реализация пути:** Использовать проверку платформы для формирования корректного пути. Кодировка файла — UTF-8.

**Формат файла (F-13):**
```json
{
  "ip": "192.168.1.10",
  "port": 1521,
  "username": "admin",
  "service_name": "ORCL"
}
```
*Примечание:* Поле `port` всегда сериализуется как целое число (integer). При десериализации модуль должен корректно обрабатывать приведение строки к `int`, если файл был отредактирован вручную.

**Обработка ошибок при чтении (F-9):**

| Ситуация | Действие |
|----------|----------|
| Файл отсутствует | Новый файл не создаётся. Поля заполняются значениями по умолчанию. |
| Файл повреждён или содержит некорректный JSON | Файл удаляется. Поля заполняются значениями по умолчанию. |
| Некорректный тип данных (например, port = «abc») | Файл удаляется. Поля заполняются значениями по умолчанию. |
| Ошибки файловой системы (нет прав на чтение) | Обрабатываются молча. Поля заполняются значениями по умолчанию. |

**Значения по умолчанию для отсутствующих в JSON полей:**

| Поле | Значение по умолчанию |
|------|----------------------|
| `ip` | Пустая строка `""` |
| `port` | `1521` |
| `username` | Пустая строка `""` |
| `service_name` | `"ORCL"` |

**Умный фокус (NF-1):** После загрузки данных модуль устанавливает фокус ввода на поле «Пароль», если поле `ip` или `username` не пустое (заполнено из файла или передано как default). В противном случае (первый запуск, все поля пусты) фокус устанавливается на поле «IP-адрес/хост».

#### 4.1.5. Сохранение параметров (F-7)
При нажатии «Ок» модуль должен сохранять IP-адрес/хост, порт, имя пользователя и идентификатор службы в JSON-файл по указанному выше пути. Пароль не сохраняется.

**Требование к атомарности записи:** Для гарантии атомарности операции на всех ОС, временный файл (`.tmp`) должен создаваться строго в той же директории, что и целевой файл `connection.json`, перед использованием стандартной функции Python `os.replace(src, dst)`. Эта функция гарантирует атомарную замену на POSIX-системах и корректно выполняет замену существующего файла на Windows, не требуя использования сторонних библиотек (ctypes).

#### 4.1.6. Возврат результата вызывающей системе
При нажатии «Ок» модуль должен возвращать вызывающей системе структуру в формате Python dict (F-5):
```json
{
  "ip": str,
  "port": int,
  "username": str,
  "password": str,
  "service_name": str
}
```
При нажатии «Отмена», закрытии окна через «X» или нажатии Esc модуль должен возвращать сигнал отмены (`None`) (F-6).

#### 4.1.7. Маскировка пароля (F-11)
При вводе в поле «Пароль» символы должны отображаться в виде символов-заменителей (например, «•»).

### 4.2. Требования к интеграции (F-12, NF-5)
Модуль предоставляет вызывающей системе API. В зависимости от режима интеграции:

1. **Режим библиотеки (по умолчанию):**
   ```python
   def show_dialog(
       default_ip: str = "",
       default_port: int = 1521,
       default_username: str = "",
       default_service_name: str = "ORCL",
       debug: bool = False
   ) -> Union[dict, None]
   ```
   *Входные параметры* — значения по умолчанию для полей.  
   *debug* — если `True`, ошибки логируются в `stderr` (без пароля).  
   *Возврат* — словарь с данными или `None` при отмене.

2. **Режим отдельного процесса (subprocess):**  
   Вызывающая система запускает исполняемый файл модуля: `subprocess.run(['login_dialog.exe'], capture_output=True, text=True)`  
   Модуль выводит результат в `stdout` в формате JSON:
   * **Успех:** `{"status": "success", "data": {"ip": "...", "port": 1521, ...}}`
   * **Отмена:** `{"status": "cancelled"}`
   * **Ошибка (критическая):** `{"status": "error", "message": "..."}` (код возврата процесса `!= 0`)

Модуль не должен иметь скрытых зависимостей от драйверов Oracle или сторонних библиотек, требующих отдельной установки (NF-7).

### Приложение Г. Пример команды сборки PyInstaller
Для корректной упаковки модуля в `.exe` с поддержкой тем CustomTkinter использовать следующую команду:

```bash
pyinstaller --onefile \
  --hidden-import customtkinter \
  --add-data "customtkinter/assets:customtkinter/assets" \
  --name login_dialog \
  src/alphameterqc/login_dialog/__main__.py
```

**Пояснение аргументов:**
- `--onefile` — упаковать всё в один `.exe` файл.
- `--hidden-import customtkinter` — явно включить модуль CustomTkinter (иначе PyInstaller может его пропустить).
- `--add-data "customtkinter/assets:customtkinter/assets"` — включить папку с темами и ресурсами CustomTkinter (критически важно для отображения UI).
- `--name login_dialog` — имя выходного файла.
- Последний аргумент — точка входа в модуль.

**Для Windows:** разделитель в `--add-data` — точка с запятой (`;`):
```bash
pyinstaller --onefile --hidden-import customtkinter --add-data "customtkinter/assets;customtkinter/assets" --name login_dialog src/alphameterqc/login_dialog/__main__.py
