# Пользовательские истории (User Stories)

**Версия:** 2.3 (итоговая)  
**Дата:** 2026-06-04  
**Автор:** Солодюк В.Л.  
**Проект:** ПО «AlphaMeterQC» / Модуль ввода идентификационных данных для подключения к БД

---

## Бэклог пользовательских историй (6 историй)

| ID | История (формат Connextra) | Критерии приемки (Given-When-Then) |
|----|---------------------------|-------------------------------------|
| US-1 | Как инженер отдела системного сопровождения, я хочу видеть окно входа с полями для IP-адреса/хоста, порта, имени пользователя, пароля и идентификатора службы на русском языке, чтобы быстро и понятно вводить данные для подключения к БД. | 1. Given модуль запущен, When окно открыто, Then видны поля «IP-адрес/хост», «Порт», «Имя пользователя», «Пароль», «Идентификатор службы (SID/Service Name)» и кнопки «Ок», «Отмена».<br>2. Given любое поле некорректно или все поля пусты, When окно открыто, Then кнопка «Отмена» всегда активна.<br>3. Given модуль использует графическую библиотеку CustomTkinter, When окно отображено, Then интерфейс имеет современный внешний вид (скругленные углы, поддержка темной темы). |
| US-2 | Как администратор БД Oracle, я хочу чтобы модуль проверял корректность вводимых параметров (IP, порт, имя, пароль, идентификатор службы) до отправки запроса в сеть, чтобы снизить количество ошибочных попыток подключения и нагрузку на сервер БД. | 1. Given IP-адрес некорректен (например, «300.1.1.1» или содержит недопустимые символы), When поле изменено, Then поле подсвечивается, кнопка «Ок» неактивна.<br>2. Given IP-адрес является валидным DNS-именем (например, «oracle-db.company.local» или «localhost»), When поле изменено, Then поле считается валидным (поддержка DNS-имен).<br>3. Given порт вне диапазона 1–65535, When проверка выполнена, Then кнопка «Ок» неактивна.<br>4. Given имя пользователя длиннее 30 символов, When проверка выполнена, Then поле подсвечивается, кнопка «Ок» неактивна.<br>5. Given имя пользователя длиной до 30 символов, When остальные поля корректны, Then имя считается валидным.<br>6. Given идентификатор службы содержит недопустимые символы или длиннее 30 символов, When проверка выполнена, Then поле подсвечивается, кнопка «Ок» неактивна.<br>7. Given все поля корректны (IP — валидный IPv4 или DNS, порт 1–65535, имя 1–30 символов, идентификатор службы 0–30 символов (допустимые символы: латиница, цифры, `_`, `.`, `-`), пароль не пустой), When проверка пройдена, Then кнопка «Ок» активна.<br>8. Given форма невалидна (кнопка «Ок» неактивна), When пользователь нажимает Enter, Then действие игнорируется, окно не закрывается. |
| US-3 | Как инженер отдела системного сопровождения, я хочу чтобы модуль сохранял IP, порт, имя пользователя и идентификатор службы между сеансами и автоматически подставлял их при следующем запуске, чтобы сократить ручной ввод с 30 до 5 секунд. | 1. Given пользователь подтвердил корректные данные, When сеанс завершён, Then IP, порт, имя пользователя и идентификатор службы сохраняются в локальный файл (пароль не сохраняется) по пути `%LOCALAPPDATA%\alphameterqc\connection.json` (Windows) или `~/.config/alphameterqc/connection.json` (Linux).<br>2. Given модуль запущен повторно, When окно открыто, Then поля IP, порт, имя пользователя и идентификатор службы заполнены сохранёнными значениями, поле пароля пустое.<br>3. Given пользователь изменил автоподставленные значения, When поля изменены, Then новые значения валидируются по правилам US-2.<br>4. Given данные успешно загружены из файла, When окно отображено, Then фокус ввода устанавливается на поле «Пароль» (для ускорения ввода), если поле `ip` или `username` не пустое.<br>5. Given файл отсутствует (первый запуск) или все поля пусты, When окно отображено, Then фокус ввода устанавливается на поле «IP-адрес/хост».<br>6. Given сохранение выполняется, When операция записи файла, Then временный файл (.tmp) создается строго в той же директории, что и целевой файл, и выполняется атомарная замена через `os.replace(src, dst)` (для гарантии атомарности на всех ОС). |
| US-4 | Как команда разработки планируемого настольного ПО, я хочу получать от модуля чёткий результат (данные или сигнал отмены) через простой интерфейс, чтобы легко интегрировать модуль в основное ПО. | 1. Given нажата кнопка «Ок» при корректных данных, When модуль завершён, Then вызывающей системе возвращается структура с полями: ip, port, username, password, service_name.<br>2. Given нажата кнопка «Отмена», закрыто окно через «X» или нажат Esc, When модуль завершён, Then вызывающей системе возвращается сигнал отмены (None или Cancel).<br>3. Given все поля корректны и нажат Enter в любом поле ввода, When Enter обработан, Then это действие равносильно нажатию кнопки «Ок».<br>4. Given модуль упакован в .exe через PyInstaller, When он запущен на целевой машине без установленного Oracle Client, Then модуль работает корректно.<br>5. Given вызывающая система использует графический фреймворк, отличный от Tkinter/CustomTkinter (например, PyQt), When модуль вызывается через subprocess, Then результат передается через stdout в формате JSON: `{"status": "success", "data": {...}}` или `{"status": "cancelled"}` или `{"status": "error", "message": "..."}`.<br>6. Given модуль работает в режиме библиотеки (по умолчанию), When вызывающая система импортирует функцию `show_dialog()`, Then интеграция происходит на уровне кода без конфликтов графических циклов (Event Loop). |
| US-5 | Как отдел информационной безопасности, я хочу чтобы модуль обеспечивал безопасную работу с паролем: не сохранял его на диск, не передавал в логи и маскировал при вводе, чтобы исключить компрометацию учётных данных. | 1. Given поле «Пароль» активно, When пользователь вводит символы, Then они отображаются в виде маскирующих символов (например, «•»).<br>2. Given пароль введён, When модуль сохраняет настройки, Then пароль не записывается на диск.<br>3. Given модуль работает, When выполняется любое действие (ввод, проверка, завершение), Then пароль не выводится в консоль, логи или внешние файлы. |
| US-6 | Как инженер отдела системного сопровождения, я хочу чтобы модуль продолжал работу без ошибок и без показа сообщений пользователю, если файл сохранённых данных повреждён или недоступен для записи, чтобы процесс подключения не прерывался. | 1. Given файл сохранения повреждён, When модуль запущен, Then повреждённый файл удаляется, новый файл не создаётся, поля заполняются значениями по умолчанию (Порт = 1521, Идентификатор службы = ORCL, остальные пустые), кнопка «Ок» неактивна, ошибки не показываются.<br>2. Given файл сохранения отсутствует, When модуль запущен, Then новый файл не создаётся, поля заполняются значениями по умолчанию (Порт = 1521, Идентификатор службы = ORCL, остальные пустые), кнопка «Ок» неактивна, ошибки не показываются.<br>3. Given у пользователя нет прав на запись в директорию сохранения, When модуль пытается сохранить данные, Then модуль продолжает работу без вывода ошибки, просто не сохраняя изменения.<br>4. Given поле `port` в JSON содержит строку (например, `"1521"`), When модуль читает файл, Then значение корректно приводится к типу integer. |
