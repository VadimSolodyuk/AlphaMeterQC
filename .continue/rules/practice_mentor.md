## 1. Последовательность работы

**Не переходи к следующему пункту отчёта без команды пользователя.** После завершения работы над текущим разделом (или подразделом) отчёта — остановись и дождись указаний. Не начинай писать следующий раздел самостоятельно.

## 2. Принцип «сначала сделай — потом напиши»

Прежде чем описывать в отчёте выполнение работы, создай соответствующие артефакты (документы, схемы, скрипты, диаграммы). Отчёт должен отражать реально выполненные действия.

Это значит:
1. Проектируешь БД — создай реальную ER-диаграмму (plantuml), продумай сущности, атрибуты, связи
2. Реализуешь БД — напиши реальные SQL-скрипты (`CREATE TABLE`, индексы, constraints)
3. Тестируешь — приведи конкретные тестовые запросы и их результаты
4. Администрируешь — опиши конкретные настройки бэкапа, скрипты, права доступа

Все созданные артефакты оформляй в едином стиле и сохраняй в `docs/specs/db`.

Текст отчёта пиши от первого лица множественного числа («была спроектирована», «разработана», «проведено тестирование») — как отчёт о реально выполненной работе.

## 3. Основные принципы содержания отчёта

- Контекст решаемой в ходе практики задачи — в `docs/practice/context_description.md`
- Отчёт должен соответствовать рабочей программе практики (`docs/practice/practice_program.txt`)
- Стилистика изложения — по примеру `docs/practice/report_text_example.txt`
- Каждый раздел отчёта — содержательный, без «воды»

## 4. Релевантные принципы из профильных /rules

### Из Business Analyst (анализ требований)
- Требования — чётко, недвусмысленно, со ссылками на источники
- Каждая сущность и атрибут должны быть обоснованы бизнес-процессом
- Фиксируй источники данных и стейкхолдеров

### Из Architect (проектирование БД)
- Нормализация данных (каждая часть данных в одном месте — SSOT)
- ER-диаграмма — основа концептуальной модели
- Первичные и внешние ключи — явно, связи — с каскадными правилами
- Индексы — осознанно, под нагрузку, а не «на всякий случай»

### Из Developer Mentor (реализация)
- SQL-скрипты — читаемые, с комментариями «почему», а не «что»
- Обработка ошибок (никаких пустых `EXCEPT` / `WHEN OTHERS THEN NULL`)
- Fail Fast: проверяй входные данные на границах

### Из QA (тестирование)
- Для каждого запроса — 2-3 граничных случая (пустая таблица, NULL, дубликаты)
- Нагрузочное тестирование — с реалистичными объёмами данных
- Проверь ссылочную целостность при удалении/обновлении

### Из DevOps (администрирование)
- Резервное копирование — стратегия (что, когда, как восстанавливать)
- Контроль доступа — минимум привилегий (POLP)
- Документация по развёртыванию БД — в отчёт

## 5. Оформление ссылок на артефакты

Ссылки на артефакты в отчёте по практике оформляй в формате:

```
(Приложение {буква} "{Название документа}", раздел {номер})
```

Где:
- Приложение А, Б, В... — сквозная нумерация всех артефактов
- Название документа — без расширения файла, в кавычках
- Раздел — номер раздела/подраздела в документе (если применимо)

Примеры:
- (Приложение А "Требования заинтересованных сторон", раздел 1)
- (Приложение Б "Анализ бизнес-процессов предметной области", раздел 2.1)

Не используй пути к файлам артефактов (path/to/file.md) в тексте отчёта — только ссылки на приложения.

## 6. Реестр соответствия приложений и артефактов

Веди файл `docs/practice/APPENDIX_REGISTRY.md`, в котором фиксируется соответствие между буквенными обозначениями приложений в отчёте (Приложение А, Б, В...) и файлами артефактов.

Правила ведения реестра:
1. **При создании нового артефакта** — сразу добавляй запись в реестр с указанием буквы приложения, названия документа и пути к файлу.
2. **Буквы приложений** — сквозная нумерация по алфавиту (А, Б, В, Г, Д, Е, Ж, И, К, Л, М, Н, П, Р, С, Т, У, Ф, Х, Ц, Ш, Щ, Э, Ю, Я).
3. **При удалении или переименовании артефакта** — актуализируй запись в реестре.
4. **Перед каждым новым разделом отчёта** — сверься с реестром, чтобы избежать дублирования букв.
5. **Формат записи**: `| {буква} | {Название документа} | {путь к файлу} | {краткое описание} |`

Файл реестра должен содержать:
- заголовок с описанием назначения;
- таблицу соответствия со столбцами: `Приложение`, `Название документа`, `Файл`, `Описание`;
- историю изменений (кто, когда, что добавил/изменил).

## 7. Запрет на растровые изображения диаграмм

Не вставляй в отчёт растровые изображения диаграмм, если они не сгенерированы. Вместо этого ссылайся на PlantUML-диаграммы в соответствующих приложениях.

## 8. Безопасное редактирование файлов

Правила безопасного редактирования файлов вынесены в отдельное правило с наивысшим приоритетом — см. `.continue/rules/edit-safety-rules.md`. **Ознакомься с ним перед любыми изменениями файлов.**

## 9. Модульная структура отчёта (разделы в отдельных файлах)

Каждый раздел отчёта хранится в отдельном файле в директории `docs/practice/report_sections/`. Файлы имеют расширение `.txt` (не `.md`), так как итоговый отчёт собирается в единый документ в формате plain text.

**Структура файлов:**

| Файл | Содержание |
|:-----|:-----------|
| `00_introduction.txt` | ВВЕДЕНИЕ. Общие сведения о практике и цели отчёта |
| `01_section_1.txt` | Раздел 1. Сбор и анализ требований к БД. Анализ бизнес-процессов и формирование требований |
| `01_subsection_1.txt` | Подраздел 1.1. Выявление заинтересованных сторон и сбор требований |
| `01_subsection_2.txt` | Подраздел 1.2. Анализ бизнес-процессов и определение ключевых сущностей |
| `01_subsection_3.txt` | Подраздел 1.3. Подготовка технического задания на проектирование БД |
| `02_section_2.txt` | Раздел 2. Разработка концептуальной и логической модели БД. Проектирование ER-диаграммы и нормализация данных |
| `03_section_3.txt` | Раздел 3. Реализация БД в СУБД. Создание таблиц, индексов и ограничений |
| `04_section_4.txt` | Раздел 4. Оптимизация и тестирование БД. Анализ производительности и нагрузочное тестирование |
| `05_section_5.txt` | Раздел 5. Администрирование и безопасность БД. Настройка резервного копирования и контроль доступа |
| `06_section_6.txt` | Раздел 6. Сбор исходной информации для ВКР. Подготовка материалов для выпускной квалификационной работы |
| `99_conclusion.txt` | ЗАКЛЮЧЕНИЕ. Итоги практики и выводы по выполненным задачам |

**Правила работы с модульной структурой:**

1. **Работай только над одним файлом раздела за раз.** Не загружай в контекст соседние разделы — вся информация уже есть в артефактах (Приложения А, Б, В...).

2. **Логические блоки внутри раздела выделяются в отдельные подпункты.** Если раздел содержит несколько логически завершённых тем, каждая из них оформляется как отдельный файл-подпункт с номером подраздела (например, `01_subsection_1.txt`, `01_subsection_2.txt`, `01_subsection_3.txt`). Это обеспечивает модульность и упрощает редактирование отдельных частей отчёта.

3. **Ссылки на артефакты** — используй формат, описанный в разделе 5. Не дублируй содержимое артефактов в текст раздела.

4. **Сборка единого отчёта** — после наполнения всех разделов выполни:
   ```bash
   bash docs/practice/build_report.sh
   ```
   Скрипт собирает `docs/practice/report.txt` из всех секций, генерирует содержание.

5. **Перед сборкой** — убедись, что реестр приложений (`docs/practice/APPENDIX_REGISTRY.md`) актуален.

6. **Нумерация рисунков** — сквозная в пределах раздела (Рисунок 1.1, 1.2, 2.1, 2.2...). При переносе текста между файлами разделов нумерация не сбивается, так как каждый раздел — отдельный документ.

7. **При завершении работы над разделом** — остановись и дождись команды пользователя перед переходом к следующему разделу (см. раздел 1).

## 10. Правила оформления текста отчёта

При написании текста отчёта соблюдай следующие правила оформления перечислений.

### 10.1. Каждый пункт перечисления — с новой строки

Каждый элемент любого перечисления начинается с новой строки (со знака переноса строки). Пункты не разделяются пустой строкой — между ними только перенос строки.

### 10.2. Типы перечислений

**а), б), в)...** — подчинённые пункты после двоеточия, детализация, уточнение. Используются для раскрытия предыдущего тезиса.

*Исключение:* буквы ё, з, й, о, ч, ь, ы, ъ не используются (аналогично правилу для букв приложений).

Пример:
```
В ходе анализа были выделены следующие группы требований:
а) функциональные требования — определяют поведение системы;
б) нефункциональные требования — определяют качественные характеристики;
в) ограничения — накладывают рамки на реализацию.
```


**1), 2), 3)...** — последовательные шаги, порядок действий.

Пример:
```
Процесс проектирования БД включал следующие этапы:
1) анализ предметной области и сбор требований;
2) построение концептуальной модели (ER-диаграмма);
3) нормализация и приведение к логической модели.
```

**—** — равнозначные перечисления. Используются, когда важен порядок следования или элементы равноправны.

### 10.3. Отсутствие пустых строк между абзацами

Между абзацами и пунктами перечислений **не должно быть пустых строк** — разделение выполняется только переносом строки. Пустая строка допустима только перед заголовками разделов.

### 10.4. Запрет markdown-разметки в файлах .txt

Файлы секций отчёта имеют расширение `.txt` и собираются в единый документ в формате plain text. **Не используй markdown-элементы** (`##`, `**...**`, `*...*`, `###` и т.п.) в этих файлах. Оформление заголовков и выделений выполняется только средствами plain text:
- заголовки разделов — номер и название (например, `1.3 Подготовка технического задания`);
- подпункты 3-го порядка — номер и название (например, `1.3.1 Обоснование выбора СУБД.`);
- выделение смысловых блоков — точка в конце заголовка и переход на новую строку.

### 10.5. Оформление подпунктов 3-го порядка

Логические подпункты внутри раздела (например, «Обоснование выбора СУБД», «Связь с предшествующими артефактами») оформляются как подпункты 3-го порядка с номером вида `X.Y.Z`:
1) **Номер и название** — на отдельной строке, заканчиваются точкой.
2) **Текст подпункта** — начинается со следующей строки, без отступа.

Пример:
```
1.3.1 Обоснование выбора СУБД.
В техническом задании в качестве целевой СУБД определён PostgreSQL...
```
