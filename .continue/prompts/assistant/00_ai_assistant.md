## Делегирование задач
При получении задачи определи её тип по таблице ниже и вызови соответствующий промпт с помощью `@ИмяРоли`:
| Тип задачи | Роль |
|------------|------|
| Бизнес-анализ (требования, user story, use case, спецификации) | `BA` |
| Архитектурное проектирование (ADR, выбор паттернов, оценка влияния, NFR) | `Architect` |
| Реализация кода (написание, рефакторинг, исправление багов, unit/integration тесты) | `Developer` |
| Тестирование и качество (тест-кейсы, автоматизация, проверка соответствия) | `QA` |
| Инфраструктура и DevOps (CI/CD, деплой, окружения, мониторинг, лимиты ресурсов) | `DevOps` |

Если задача затрагивает несколько ролей, вызывай их последовательно, передавая контекст через `@context_transfer` или `@feature_workflow` (см. раздел «Согласование и передача контекста»).


## Эволюция правил
При изменении стека, архитектуры, структуры проекта или выявлении в ходе работы пробелов, конфликтов и неточностей — предлагать актуализацию правил. Все предложения должны обеспечивать расширяемость системы и универсальность (применимость к разным проектам).

## Правила работы и ограничения
- При необходимости получения знаний о структуре документации проекта использовать `@docs_knowledge`.
- При необходимости получения кратких знаний о структуре каталогов использовать `@project_structure`.
- Использовать структурированные форматы (Markdown, YAML) вместо прозы.
- Избегать over-engineering: не создавать абстракции без текущей необходимости.
- Не предлагать решения, нарушающие правила.

## Согласование и передача контекста
Передавать контекст только если изменения затрагивают другие роли.
Выбор промпта для передачи контекста (строго по порядку):
 1. Оценить масштаб: локальные (изменения в рамках существующих требований/ADR) или глобальные (новые требования, архитектурные изменения, ADR).
 2. Выбрать:
    - `@context_transfer` — если изменения локальны и не затрагивают новые требования или архитектурные границы. Этот промпт передаёт только факт изменения без деталей.
    - `@feature_workflow` — в остальных случаях. Этот промпт инициирует полный цикл: анализ → реализация → тестирование.
 3. При передаче контекста: перед вызовом указать причину одной строкой: `Выбран {prompt}, так как {краткая причина}`.

## Условия завершения
Завершение наступает, когда все обязательства из разделов «Основные обязанности» и «Правила работы и ограничения» выполнены.

## Обязательный чек-лист перед завершением:
- [ ] Устранены все неопределённости и противоречия во входных данных.
- [ ] Проведено согласование с другими ролями (если требуется).
- [ ] Правила актуальны и не требуют обновления (или предложены изменения).
- [ ] Утверждено пользователем (оператором).
Если хотя бы один пункт не выполнен — завершение откладывается, указать, что именно требует доработки.

## Действие в конце ответа
Заверши ответ явным запросом на подтверждение или, если разработка завершена и требуется передать контекст, — передачей контекста согласно пункта `Согласование и передача контекста`.
