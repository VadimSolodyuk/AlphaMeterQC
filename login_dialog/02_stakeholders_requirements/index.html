Требования заинтересованных сторон

Модуль ввода идентификационных данных для подключения к БД

Версия документа: 2.5 (итоговая)
Дата: 2026-06-04
Автор: Бизнес-аналитик
Проект: ПО «AlphaMeterQC» / Модуль ввода идентификационных данных для подключения к БД

Термины и определения

Термин Определение
Система Разрабатываемый программный модуль ввода идентификационных данных для подключения к БД.
Идентификационные данные Совокупность параметров, необходимых для подключения к серверу БД Oracle: IP-адрес/хост, порт, имя пользователя, пароль, идентификатор службы (SID/Service Name).
Файл сохранённых данных Локальный файл в защищённой области пользователя (%LOCALAPPDATA%\alphameterqc\connection.json для Windows или ~/.config/alphameterqc/connection.json для Linux), в котором система сохраняет IP-адрес/хост, порт, имя пользователя и идентификатор службы между сеансами. Пароль не сохраняется.
Правила проверки Набор формальных правил, определяющих корректность введённых данных (см. справочную таблицу ниже).
Режим библиотеки Режим интеграции, при котором модуль импортируется как Python-пакет и вызывается напрямую.
Режим отдельного процесса Режим интеграции, при котором модуль запускается как самостоятельный процесс (subprocess), обмен данными происходит через stdout (JSON). Используется для исключения конфликтов графических циклов (Event Loop).

Заинтересованные стороны

ID Заинтересованное лицо/сторона Описание
S-1 Инженеры отдела системного сопровождения Основные конечные пользователи модуля. Ежедневно вводят идентификационные данные для подключения к БД.
S-2 Руководитель отдела системного сопровождения Внутренний заказчик. Отвечает за эффективность работы отдела, KPI по времени настройки и количеству ошибок подключения.
S-3 Команда разработки планируемого настольного ПО Технические специалисты, которые будут интегрировать модуль в основное приложение.
S-4 Администраторы БД Oracle Специалисты, обеспечивающие работу серверов БД. Заинтересованы в снижении количества ошибочных попыток подключения.
S-5 Отдел информационной безопасности Контролирующий орган. Следит за соблюдением политик хранения и передачи идентификационных данных.

Требования заинтересованных сторон

1. Отображение графического интерфейса

ID Заинтересованное лицо Версия Содержание требования
SR-1 S-1 v3 Все подписи элементов управления должны быть выполнены на русском языке: «IP-адрес/хост», «Порт», «Имя пользователя», «Пароль», «Идентификатор службы (SID/Service Name)», «Ок», «Отмена». Графический интерфейс должен быть реализован с использованием библиотеки CustomTkinter для обеспечения современного внешнего вида.

2. Валидация данных и управление кнопкой «Ок»

ID Заинтересованное лицо Версия Содержание требования
SR-2 S-4 v2 Система должна выполнять предварительную проверку формата IP-адреса/хоста (IPv4 или DNS-имя), порта и идентификатора службы до отправки любого запроса к серверу БД.
SR-3 S-1, S-4, S-5 v3 Система должна визуально выделять поле ввода, содержимое которого не соответствует правилам проверки, в реальном времени при изменении содержимого.
SR-4 S-1, S-4 v2 Кнопка «Ок» должна быть неактивна до тех пор, пока все обязательные поля не заполнены и не соответствуют правилам проверки. Кнопка «Отмена» должна быть активна всегда.

3. Автоподстановка и работа с файлом сохранённых данных

ID Заинтересованное лицо Версия Содержание требования
SR-5 S-1 v3 Система должна при каждом запуске автоматически подставлять в соответствующие поля ранее сохранённые значения IP-адреса/хоста, порта, имени пользователя и идентификатора службы. При отсутствии файла или его повреждении поля заполняются значениями по умолчанию (порт = 1521, идентификатор службы = ORCL, остальные пустые). Поле пароля должно оставаться пустым.
SR-6 S-1 v2 Система должна обеспечивать стабильную работу без зависаний при вводе и проверке параметров подключения. Время отклика интерфейса не должно превышать 0,1 секунды.
SR-7 S-1, S-3 v2 При чтении файла сохранённых данных система должна обрабатывать следующие ситуации без вывода сообщений пользователю:
— при отсутствии файла — не создавать новый, поля заполняются значениями по умолчанию;
— при повреждении файла — удалить его, поля заполняются значениями по умолчанию;
— при любой ошибке файловой системы — продолжать работу, поля заполняются значениями по умолчанию.

4. Передача данных и бизнес-эффективность

ID Заинтересованное лицо Версия Содержание требования
SR-8 S-2 v2 Система должна обеспечивать снижение среднего времени ввода идентификационных данных с 30 до 5 секунд на одно подключение.
SR-9 S-2 v3 Система должна снизить до нуля количество ошибок подключения к БД, вызванных неверным форматом вводимых данных (IP-адрес/хост, порт, пустые поля, идентификатор службы).
SR-10 S-2, S-3 v3 Система должна предоставлять графический интерфейс для ввода всех параметров подключения. Модуль должен поддерживать два режима интеграции: режим библиотеки (по умолчанию) и режим отдельного процесса (для исключения конфликтов графических циклов с вызывающей системой). Интеграция не должна требовать от основного ПО изменения кода или перекомпиляции.
SR-11 S-3 v2 Система должна предоставлять чёткий и задокументированный контракт взаимодействия (API) для вызова модуля и получения идентификационных данных.

5. Интеграция и технические ограничения

ID Заинтересованное лицо Версия Содержание требования
SR-12 S-3 v2 Система не должна иметь скрытых зависимостей от драйверов Oracle или сторонних библиотек, требующих отдельной установки. Допускается использование CustomTkinter как единственной внешней зависимости (упаковывается вместе с приложением).
SR-13 S-3 v2 Система должна быть совместима с упаковкой в один исполняемый файл (.exe) с помощью PyInstaller.

6. Требования информационной безопасности

ID Заинтересованное лицо Версия Содержание требования
SR-14 S-5 v2 Система не должна сохранять пароль на диске в открытом виде или в любом другом виде, допускающем его восстановление.
SR-15 S-5 v2 Поле ввода пароля в графическом интерфейсе должно быть замаскировано (отображать символы-заменители).
SR-16 S-5 v2 Система не должна передавать пароль в текстовом виде в логи, консоль, внешние файлы или любые другие внешние источники.

Правила проверки корректности полей (справочно)

Поле Правило проверки
IP-адрес / Хост Валидный IPv4-адрес (0-255) или DNS-имя хоста. Для DNS: строка после удаления пробелов по краям (strip) должна содержать хотя бы один буквенно-цифровой символ, не начинаться и не заканчиваться на '-' или '.', допустимые символы: латиница, цифры, '_', '-', '.', не более 255 символов.
Порт Число от 1 до 65535.
Имя пользователя Не пустое, не более 30 символов (латиница, цифры, '_', '-').
Пароль Не пустой (хотя бы 1 символ, любые символы).
Идентификатор службы (SID/Service Name) Не более 30 символов, допустимые символы: латиница, цифры, '_', '.', '-'. Поле опциональное (может быть пустым).

Соответствие критериям приёмки Концепции (раздел 10)

Критерий Концепции Покрывающие требования ТЗС
1. Интеграция без изменения кода SR-10, SR-11
2. Пять полей и кнопки SR-1
3. Кнопка «Ок» неактивна при ошибках SR-4
4. Сохранение и возврат данных при «Ок» SR-5, SR-11, SR-14
5. Заполнение полей из файла при вызове SR-5, SR-7
6. Сигнал отмены SR-11
7. Маскировка пароля SR-15
8. Пароль не передаётся в логи SR-16
9. Ошибки ФС обрабатываются молча SR-7
10. Окно отображается < 0,5 с SR-6
