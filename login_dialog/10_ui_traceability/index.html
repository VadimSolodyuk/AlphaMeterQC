Матрица трассировки элементов UI и требований

Версия: 1.0
Дата: 07.06.2026 14:00:GMT+5
Автор: Солодюк В.Л.
Проект: ПО «AlphaMeterQC» / Модуль ввода идентификационных данных для подключения к БД
Домен: Проектирование UI/UX

1. Введение

1.1 Цель документа

Подтвердить, что спроектированный пользовательский интерфейс (макет 09_ui_design.md и прототип prototype.py) полностью покрывает функциональные и нефункциональные требования Спецификации требований (SRS v2.9).

1.2 Область применения

Документ предназначен для QA-инженеров (для написания тест-кейсов) и разработчиков (как чек-лист при реализации бизнес-логики UI).

2. Матрица соответствия UI и функциональных требований (F)

ID требования Описание требования (SRS) Элемент UI / Поведение прототипа Статус покрытия
F-1 Отображать окно с 5 полями и кнопками (CustomTkinter). Подписи на русском. Окно LoginDialogPrototype (450x420). 5 полей CTkEntry с лейблами CTkLabel. Кнопки CTkButton «Ок» и «Отмена». ✅ Полное
F-2 Проверка корректности полей в реальном времени (IPv4 + строгий DNS). Событие KeyRelease на полях ввода. Вызов метода Validator.validateAll() (будет реализован на Шаге 2). ⏳ Запланировано
F-3 Визуально подсвечивать поле, не соответствующее правилам. Изменение цвета рамки поля (border_color) на красный (#FF4444) при ошибке валидации. ⏳ Запланировано
F-4 Кнопка «Ок» неактивна при ошибках. Enter игнорируется при невалидной форме. Свойство state=ctk.DISABLED для кнопки «Ок». Обработчик bind("<Return>") проверяет флаг валидации перед закрытием. Запланировано
F-5 Возврат структуры {ip, port, username, password, service_name} при «Ок». Метод _on_ok() (сейчас заглушка с print, будет возвращать dict в реальном коде). ⏳ Запланировано
F-6 Возврат сигнала отмены (None) при «Отмена», Esc или закрытии окна. Метод _on_cancel(). Обработка protocol("WM_DELETE_WINDOW") и клавиши Escape. Запланировано
F-7 Атомарное сохранение параметров (кроме пароля) через os.replace. Бэкенд-логика. Триггерится кнопкой «Ок». В UI не отображается, но блокирует закрытие окна до завершения (если потребуется). Запланировано
F-8 Подстановка параметров из файла при запуске + умный фокус. Метод __init__ вызывает ConnectionConfig.loadFromFile(). Логика setSmartFocus() (фокус на Пароль, если ip или username не пустые). ⏳ Запланировано
F-9 Обработка отсутствия/повреждения файла (значения по умолчанию). Метод __init__. При ошибке чтения JSON поля заполняются: Порт="1521", Сервис="ORCL". Запланировано
F-10 Отсутствие прав на запись (молчаливая обработка). Бэкенд-логика. Триггерится кнопкой «Ок». UI не выводит messagebox.showerror. ⏳ Запланировано
F-11 Маскировка вводимого пароля символами-заменителями. Свойство show="•" у поля password_entry. ✅ Полное (в прототипе)
F-12 Контракт взаимодействия (библиотека/subprocess). Архитектурное решение. Реализуется в классе LoginDialog (обертка над UI). ⏳ Запланировано
F-13 JSON-формат, пути %LOCALAPPDATA% / ~/.config/, port как int. Бэкенд-логика. Класс ConnectionConfig. UI только отображает строковые значения. ⏳ Запланировано

3. Матрица соответствия UI и нефункциональных требований (NF)

ID требования Описание требования (SRS) Реализация в UI / Архитектуре Статус
NF-1 Умный фокус (снижение времени ввода). Метод setSmartFocus(): если ip или username не пустые → password_entry.focus_set(). ⏳ Запланировано
NF-2b Нет зависаний >0,5 с. Валидация выполняется синхронно и быстро (regex). Файловые операции (чтение/запись) вынесены в отдельные методы, не блокирующие mainloop. Запланировано
NF-3a Пароль не сохраняется на диск. Метод ConnectionConfig.saveAtomically() явно исключает поле password из сериализации в JSON. ⏳ Запланировано
NF-3b Пароль не передаётся в логи/консоль. В коде прототипа и будущем коде запрещено использование print(self.password_entry.get()) и логирование пароля. ✅ Полное (правило)
NF-4 Открытие окна ≤ 0,5 с. Минимизация импортов в __init__. CustomTkinter инициализируется лениво. ⏳ Запланировано
NF-7 Упаковка в .exe (PyInstaller). Использование только стандартной библиотеки + customtkinter. Отсутствие зависимостей от ОС (кроме путей к файлам). ✅ Полное (архитектура)

4. Чек-лист для QA (Смоук-тесты UI)

На основе данной матрицы тестировщик может выполнить следующие проверки без написания кода (на этапе прототипа):

  1. [ ] Запустить prototype.py. Окно открывается, размер 450x420, заголовок «Подключение к БД».
  2. [ ] Все 5 полей имеют подписи на русском языке.
  3. [ ] В поле «Пароль» вводимые символы маскируются точками.
  4. [ ] По умолчанию в поле «Порт» стоит «1521», в «Идентификатор службы» — «ORCL».
  5. [ ] Кнопка «Ок» серая (неактивна), кнопка «Отмена» активна.
  6. [ ] Нажатие Tab перемещает фокус по порядку: IP → Порт → Имя → Пароль → SID → Ок → Отмена.
  7. [ ] Нажатие Esc или клик на «Отмена» закрывает окно.
