Модуль ввода идентификационных данных для подключения к БД

Версия: 3.4 (итоговая)
Дата: 04.06.26/текущее время
Автор: В.Л. Солодюк
Проект: ПО «AlphaMeterQC» (анализ и контроль своевременности поступления, полноты и корректности данных, собранных со счётчиков электрической энергии и хранящихся в БД (Oracle) ПО АльфаЦЕНТР) / Модуль ввода идентификационных данных для подключения к БД
Статус: Утверждена

1. Название продукта / фичи

Модуль login_dialog — графический диалог для ввода идентификационных данных подключения к БД Oracle.

2. Цель создания

Разработать программный модуль login_dialog — графический интерфейс для ввода пользователем параметров подключения к БД Oracle (IP-адрес/хост, порт, имя пользователя, пароль, идентификатор службы), с функцией валидации формата вводимых данных, маскировкой пароля и последующей передачей этих данных вызывающему ПО. Модуль не выполняет самостоятельного подключения к БД, не проверяет достоверность учётных данных на сервере и не взаимодействует напрямую с модулем подключения к БД.

Бизнес-показатели успеха: - Сокращение среднего времени ввода учётных данных пользователем с 30 до 5 секунд за счёт автоподстановки сохранённых данных. - Исключение ошибок подключения к БД, вызванных неверным форматом вводимых данных (IP-адрес/хост, порт, пустые поля).

4. Целевая аудитория

  • Основные пользователи: Инженеры отдела системного сопровождения — пользователи планируемого к разработке настольного ПО.
  • Демография / сегменты аудитории: Сотрудники технических специальностей с опытом работы с базами данных, уверенные пользователи ПК. Уровень цифровой грамотности — высокий.
  • География использования: Предприятия, на которых внедрена АСКУЭ. Предполагается работа в корпоративной сети предприятия.
  • Сценарий использования: При каждом запуске ПО «AlphaMeterQC» вызывает модуль login_dialog для получения параметров подключения. Пользователь вводит или изменяет параметры, вводит пароль и подтверждает ввод. Модуль возвращает введённые данные ПО. ПО передаёт полученные данные модулю подключения к БД для установки соединения.

7. Ключевые характеристики продукта / фичи

7.1. Функциональные возможности

  • Отображение окна с полями для ввода:
  • IP-адрес / Хост
  • Порт
  • Имя пользователя
  • Пароль (с маскировкой)
  • Идентификатор службы (SID / Service Name)
  • Кнопки управления: «Ок» (подтверждение), «Отмена» (отмена).
  • Поле пароля скрывает вводимые символы (маскировка).
  • Кнопка «Ок» неактивна, пока все поля не заполнены корректно согласно правилам проверки.
  • Кнопка «Отмена» доступна всегда. При её нажатии модуль завершает работу с сигналом отмены.
  • При нажатии «Ок» модуль возвращает введённые данные вызывающей системе (структура с полями: IP, порт, имя пользователя, пароль, идентификатор службы).
  • Сохранение всех введённых данных, кроме пароля, локально в защищённой области пользователя.
  • При следующем запуске сохранённые поля автоматически подставляются в соответствующие поля ввода. Поле пароля остаётся пустым.
  • Контракт взаимодействия: Модуль предоставляет вызывающей системе публичный API для вызова и получения результата (данные или сигнал отмены). Интеграция не требует изменения кода вызывающей системы.
  • Режимы интеграции: Модуль поддерживает два режима работы для исключения конфликтов графических циклов (Event Loop):
  • Режим библиотеки (по умолчанию): Модуль импортируется как Python-пакет, функция вызывается напрямую.
  • Режим отдельного процесса (альтернативный): Модуль запускается как самостоятельный процесс (subprocess), обмен данными происходит через stdout (JSON). Этот режим гарантирует совместимость с любой вызывающей системой (PyQt, C++, C# и т.д.).
  • Безопасность: Пароль не передаётся в логи, консоль, внешние файлы или иные внешние источники.

  • Правила проверки корректности: | Поле | Правило проверки | |------|------------------| | IP-адрес / Хост | Валидный IPv4-адрес (0-255) или DNS-имя хоста. Для DNS: строка после удаления пробелов по краям (strip) должна содержать хотя бы один буквенно-цифровой символ, не начинаться и не заканчиваться на '-' или '.', допустимые символы: латиница, цифры, '', '-', '.', не более 255 символов. | | Порт | Число от 1 до 65535 | | Имя пользователя | Не пустое, не более 30 символов | | Пароль | Не пустой (хотя бы 1 символ) | | Идентификатор службы (SID/Service Name) | Не более 30 символов, допустимые символы: латиница, цифры, '', '.', '-' |

  • При невыполнении любого правила — кнопка «Ок» неактивна, поле с ошибкой визуально выделяется.

  • Модуль не выполняет подключение к БД, не проверяет доступность сервера и не вызывает внешние модули подключения.
  • Обработка файла сохранения при чтении:
  • При отсутствии файла сохранения — новый файл не создаётся, поля заполняются значениями по умолчанию (Порт = 1521, Идентификатор службы = ORCL, остальные пустые).
  • При повреждении файла сохранения — повреждённый файл удаляется, новый файл не создаётся, поля заполняются значениями по умолчанию.
  • Ошибки файловой системы при чтении обрабатываются молча, без вывода сообщений пользователю.

7.2. Нефункциональные требования

  • Язык реализации: Python 3.9 или выше.
  • Графическая библиотека: CustomTkinter (современная надстройка над стандартным Tkinter, обеспечивает современный UI, минимальные зависимости и быструю упаковку в .exe через PyInstaller).
  • Интеграция: Модуль должен быть совместим с упаковкой в исполняемый файл (.exe). Зависимость от Oracle отсутствует — модуль не требует установки драйверов БД.
  • Тайм-аут ожидания ввода: не предусмотрен. Модуль ждёт действий пользователя бесконечно.
  • Хранение данных: без шифрования, только в локальной пользовательской директории (%LOCALAPPDATA% для Windows, ~/.config/ для Linux).
  • Производительность: окно должно отображаться после вызова модуля без заметной задержки для пользователя.
  • Надёжность: модуль не должен зависать или выдавать необработанные ошибки при проблемах с файловой системой.
  • Удобство использования: все элементы управления должны быть явно видимы и подписаны на русском языке.
  • Безопасность: Пароль не передаётся в логи, консоль, файлы или иные внешние источники.

8. Уникальные преимущества (USP)

  • Целенаправленность: Модуль решает ровно одну задачу — сбор и передачу идентификационных данных для подключения к Oracle — без избыточного функционала.
  • Простота интеграции: Чёткий интерфейс взаимодействия и поддержка двух режимов (библиотека/subprocess) позволяют легко встроить модуль в любое настольное ПО без конфликтов GUI.
  • Современный UI: Использование CustomTkinter обеспечивает современный внешний вид при сохранении легковесности.
  • Безопасность по умолчанию: Пароль не сохраняется на диске, поле ввода маскируется, пароль не передаётся в логи.
  • Устойчивость к сбоям: При проблемах с чтением или записью файла сохранённых данных модуль продолжает работу без вывода ошибок пользователю.

9. Этапы реализации

Этап Описание
0 Spike (Исследование): Прототип запуска subprocess и парсинга JSON из stdout для снятия архитектурного риска.
1 Разработка графического интерфейса (CustomTkinter): Создание окна с 5 полями ввода, кнопками «Ок» и «Отмена», маскировкой пароля, подписями на русском языке.
2 Реализация проверки вводимых данных: Внедрение правил проверки для каждого поля (включая строгую поддержку DNS-имен), визуальное выделение ошибочных полей, управление доступностью кнопки «Ок».
3 Реализация сохранения и загрузки данных: Чтение и запись файла с сохранёнными данными (кроме пароля) в защищённой области пользователя, автоматическая подстановка при запуске, обработка ошибок файловой системы.
4 Реализация интерфейса для вызывающей системы: Создание точки входа для вызова модуля из основного ПО (включая поддержку режима subprocess), возврат введённых данных или сигнала об отмены/ошибки.
5 Тестирование и проверка готовности к интеграции: Проверка соответствия критериям приёмки, подготовка модуля к встраиванию.

10. Ожидаемые результаты и метрики успеха

Бизнес-метрики: - Сокращение среднего времени ввода учётных данных пользователем с 30 до 5 секунд. - Снижение количества ошибок подключения к БД, вызванных неверным форматом ввода, до нуля.

Ключевые критерии приёмки: 1. Модуль интегрируется без изменения кода вызывающей системы. 2. Окно содержит пять полей (IP/хост, порт, имя, пароль, идентификатор службы) и кнопки «Ок»/«Отмена». 3. При некорректных данных кнопка «Ок» неактивна. 4. При нажатии «Ок» модуль сохраняет параметры (кроме пароля) в файл и возвращает данные вызывающему ПО. 5. При каждом вызове поля заполняются из файла, если он существует. При отсутствии или повреждении файла поля заполняются значениями по умолчанию (Порт = 1521, Идентификатор службы = ORCL, остальные пустые). 6. При нажатии «Отмена» модуль возвращает сигнал отмены. 7. Пароль маскируется при вводе. 8. Пароль не передаётся в логи, консоль или внешние файлы. 9. Ошибки файловой системы обрабатываются молча (работа продолжается). 10. Окно отображается менее чем за 0,5 секунды.
